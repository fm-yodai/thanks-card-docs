# 開発の進め方ガイド

チーム全員で共通認識を持つための資料。

## What & Why と How

ソフトウェア開発では、大きく2つの問いに答える必要がある。

| 区分 | 問い | 内容 | 例 |
|------|------|------|-----|
| What & Why（要求定義） | 何を・なぜ作るか | プロダクトの目的、ユーザーが誰か、どんな価値を届けるか | 「全社員が誰にでもありがとうカードを送れるようにしたい」 |
| How（要件定義・骨格） | どう作るか（大まかに） | 画面構成、技術基盤、MVP全体の構造を描く | 「カード送信画面がある。宛先を選んでメッセージを書いてAIラベルが付いて送信」 |
| How（要件定義・詳細） | どう作るか（細かく） | 画面の細かい振る舞い。実装する直前に詰める粒度 | 「宛先は部署フィルタ可」「最大500文字」「AIラベルは手動変更可」 |

ウォーターフォールでは What & Why を「要求定義」、How を「要件定義」と呼ぶ。これらはウォーターフォール固有のフェーズ名であり、アジャイルでは使わない。

## アジャイルに「要求定義」「要件定義」はない

アジャイルには「要求定義」「要件定義」という工程はない。Sprint 0で要求+骨格の初版を決め、スプリントを繰り返しながら詳細化していく。

### ウォーターフォール（従来型）

要求定義 → 要件定義（骨格＋詳細）→ 設計 → 実装 → テスト → リリース

最初に全部決めて、最後にやっと動く。

### アジャイル（今回の進め方）

Sprint 0: 要求+骨格の初版を決める → Sprint 1〜N（くり返す）: プランニング（詳細を決める）→ 設計 → 実装 → テスト → 見せる → 声を聞く → 修正 → MVPリリース

2週間ごとに動くものができる。

**ポイント:** ウォーターフォールでは骨格も詳細もまとめて最初に全部やる。アジャイルでは骨格だけ先に決めて、詳細は各スプリントで都度決める。だから途中で方向修正がしやすい。

## WF → アジャイル対応表

| WF フェーズ | アジャイルでの対応 | タイミング |
|------------|-------------------|-----------|
| 要求定義（What / Why） | プロダクトゴール策定、インセプションデッキ、UX設計・プロトタイプ作成 | Sprint 0 で初版作成。以降も継続的に見直し |
| 要件定義・骨格（How の大枠） | 技術選定・環境構築、プロダクトバックログ作成、完成の定義（DoD） | Sprint 0 で初版作成。以降も継続的に見直し |
| 要件定義・詳細（How の詳細） | スプリントプランニングでバックログ項目を詳細化（スプリントバックログ作成） | 毎スプリント |
| 設計・実装・テスト | スプリント内で一貫して実施 | 毎スプリント |

## スクラムガイドが定めていること

スクラムの公式ルール（スクラムガイド 2020）が定めているのは以下の要素だけ。

- **3つの役割:** プロダクトオーナー、スクラムマスター、開発者
- **5つのイベント:** スプリント、プランニング、デイリースクラム、レビュー、レトロスペクティブ
- **3つの作成物:** プロダクトバックログ、スプリントバックログ、インクリメント
- **3つの確約:** プロダクトゴール、スプリントゴール、完成の定義（DoD）

Sprint 0、インセプションデッキ、デザインスプリント、技術選定などは、すべてチームが選ぶプラクティス。

## Sprint 0 という選択

Sprint 0 はスクラムガイドには定義されていないが、Sprint 1 を始めるための準備期間として多くのチームが採用している。

### Sprint 1 開始に必要な4つの準備（スクラムガイドから導出）

1. **スクラムチームの構成** — SM 1名 + PO 1名 + 開発者（複数名）
2. **プロダクトゴール** — プロダクトの将来の状態を示す目標
3. **プロダクトバックログ** — 優先順位付けされた作業リスト
4. **完成の定義（DoD）** — インクリメントの品質基準

### そのほかに決めておくとよいこと

- **UX 設計** — ユーザーフロー・ワイヤーフレーム・プロトタイプ
- **技術選定・環境構築** — 言語・フレームワーク・CI/CD・リポジトリ

本プロジェクトでは上記の6つすべてを実施。

## Sprint 0 でやること

| # | 決めること | 手段・プラクティス | 主導 | 区分 |
|---|----------|-------------------|------|------|
| 1 | スクラムチームの構成 | キックオフで決定済み | — | ガイド準拠 |
| 2 | プロダクトゴール | インセプションデッキ（簡略版） | 企画リード | ガイド準拠 |
| 3 | UX 設計 | デザインスプリント（簡略版） | 企画リード | 追加 |
| 4 | 技術選定・環境構築 | 技術選定（言語・FW決定）、環境構築（CI/CD・リポジトリ） | テックリード | 追加 |
| ↺ | 技術的制約を踏まえ、UX・プロトタイプを調整 | 企画リード＋テックリード協働 | — | — |
| 5 | 完成の定義（DoD） | チーム全員で策定・合意 | テックリード | ガイド準拠 |
| 6 | プロダクトバックログ | UX設計をもとにチームで洗い出し＋優先順位付け | PO＋テックリード | ガイド準拠 |

すべての工程で**POが最終意思決定**を行う。

## Sprint 0 の進め方

### 前半：企画リード主導

プロダクトゴールの定義（インセプションデッキ簡略版）→ PO合意（ゴールの最終確認）→ UX設計（デザインスプリント簡略版）→ チームレビュー（フィードバック収集）

PO・開発チームと協働し、要求を言語化・可視化する。プロダクトゴール・プロトタイプが成果物。

### 後半：テックリード（+PO）主導

技術選定（言語・FW決定）→ UX・プロト調整（技術制約の反映/企画リード協働）→ 環境構築（CI/CD・リポジトリ）→ DoD策定（完成の基準を合意）→ バックログ作成（洗い出し＋優先順位決定）

前半の成果物（プロダクトゴール・プロトタイプ）をインプットとして、要件の骨格・開発基盤を固める。

## Sprint 1 以降の進め方

2週間を1サイクル（＝スプリント）として、毎回「動くもの」を作る。

### スプリントの流れ

| タイミング | 内容 |
|-----------|------|
| Day 1 | **プランニング** — バックログから今回作る機能を選ぶ。骨格をもとに具体的な仕様を詰める。1〜2時間 |
| Day 2-12 | **開発** — 設計・実装・テストを進める。毎朝15分だけ集まって進捗と困りごとを共有 |
| Day 13-14 | **レビュー＆ふりかえり** — 作ったものを見せてFBをもらう。進め方を振り返って次のスプリントに活かす |

サイクル: 決める → 作る → 見せる → 振り返る → 繰り返し

### スプリント計画（目安）

| スプリント | 内容 |
|-----------|------|
| Sprint 1 | カードの送信・受信ができる |
| Sprint 2 | AIラベリングが付く、タイムラインで一覧表示 |
| Sprint 3 | UI改善、通知機能 |
| Sprint 4 | 月次レポート、AI精度チューニング |
| Sprint 5 | ベータテスト、フィードバック反映 |
| Sprint 6 | MVPリリース |

スプリントの内容は固定ではなく、FBや課題に応じて毎回見直す。

## 役割一覧

| 役割 | 責任 | Sprint 0 | Sprint 1〜N |
|------|------|----------|------------|
| PO | プロダクト価値の最大化。全工程で方向性の意思決定・最終承認 | 最終意思決定 | 最終意思決定 |
| 企画リード | Sprint 0 前半を主導。インセプションデッキ、UX設計・プロトタイプを作成しチームに引き継ぐ | 前半ファシリテート | 離脱 |
| テックリード | Sprint 0 後半を主導。技術選定・環境構築をリード。Sprint 1 以降も技術面の判断・方針を継続してリード | 後半ファシリテート | 技術リード継続 |
| SM | スクラムイベントのファシリテート、障害物の排除、チーム支援 | 参加者として関与 | イベントファシリテート |
| 開発メンバー | インクリメント（スプリントの成果物）の作成。Sprint 0 ではフィードバック提供・DoD策定・バックログ作成に参加。技術選定・環境構築では担当領域の推進を担う | 参加・フィードバック＋担当領域の技術推進 | 設計・実装・テスト |

## なぜアジャイルで進めるのか（補足）

1. **作らないとわからない** — AIラベリングの精度やUIの使い勝手は、動かして初めて「ここが違う」とわかる。骨格は先に決めるけど、細部は作りながら詰める方が手戻りが少ない。
2. **早く触って学ぶ** — 主目的は開発体験を通じた成長。ドキュメントを書く時間より、動くものを触る時間を増やした方が実践的なスキルが身につく。
3. **ユーザーがそばにいる** — 同じオフィスの仲間が使うアプリ。作ったものを見せてすぐ声を聞ける環境だから、フィードバックサイクルが回しやすい。

**大事な考え方:** 最初から正解を出そうとしない。最初のバージョンは粗くていい。途中で方向転換するのは失敗ではなく、学習の成果。

## デザインの進め方（補足）

「先にデザインを完璧にしてから開発」ではなく、プロダクトと一緒に育てる。

| スプリント | デザインの成長 |
|-----------|--------------|
| Sprint 0 | 最小限の基盤 — 色3〜4色、フォント、余白、ボタン |
| Sprint 1 | + カードUI — 送信画面を作る中でカードの形が決まる |
| Sprint 2 | + リスト表示 — タイムライン画面でリストが整理される |
| Sprint 3 | + グラフ表現 — レポート画面でグラフが加わる |

## 骨格の柔軟性（補足）

Sprint 0で確定した骨格は「現時点でのベストな判断」であって、絶対に守らなければいけない仕様書ではない。Sprint 1以降で実際に作ってみて見直しが必要になったら、骨格レベルの見直しをして構わない。ただし、骨格を変えるときはチーム全体で共有・合意すること。
