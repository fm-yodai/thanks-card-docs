# プロジェクト体制図

各フェーズのオーナーと、プロジェクトを横断して支える役割。

## 体制図の見方

- 各フェーズに配置されたメンバーがオーナーとして責任を持って推進する
- 配置されていないフェーズでも、学びのために参加するのは自由（「学び参加」）

## フェーズ別体制

### Phase 1: 要求定義（1〜2週間）

方向性を揃え、MVPの骨格を描く。プロジェクトチャーター作成と、UX設計・プロトタイプの初版作成まで。

**タスク:** プロジェクトチャーター作成、UX設計、プロトタイプ初版、ユーザー体験設計

**オーナー:**
- 藤田 — 全部
- 江崎 — POとして
- （川上） — 稟申のための情報収集

**学び参加:** 武田、小林、その他

### Phase 2: Sprint 0（1〜2ヵ月）

要求定義の成果物を受け取り、開発の土台をつくる。技術選定・環境構築・バックログ作成を並行して進め、MVP全体の骨格を確定させる。

**タスク:** 要件定義（骨格）、PJ管理ツール決定、PBR作成、技術選定、環境構築、インフラ構築、UX・プロトタイプ調整、デザインシステム（初版）

**オーナー:**
- 藤田 — UX・プロトタイプ調整
- 中島 — 要件定義（骨格）
- 絹田 — インフラ構築・要件定義（見学）
- まおまお — 技術選定・環境/インフラ構築
- 森本 — インフラ構築
- 中嶋 — インフラ構築
- 朴木 — インフラ構築
- 猪股 — インフラ構築
- 川上 — SMとして（稟申のための情報整理）
- 江崎 — PO・TLとして（デザインシステム初版・技術選定・稟申のための情報整理）

**学び参加:** 武田（要件定義・骨格）、小林（要件定義・骨格）

### Phase 3: 開発スプリント（Sprint 1〜N）

骨格が固まったら、2週間サイクルで機能を積み上げる。詳細な仕様はプランニングで都度決めながら実装していく。

**担当領域:**

| 領域 | メンバー |
|------|---------|
| 全体 | 川上（SM）、江崎（PO・TL）、中島（要件定義・詳細）、武田（要件定義・詳細/学び参加） |
| 生成AI組み込み | 森本、中嶋、朴木、猪股、中島 |
| データ分析フロー | 猪股、江崎 |
| 生成AIによる開発体験 | 猪股、江崎、吉田 |
| デザインシステム磨きこみ | 江崎 |
| その他実装 | 生成AI、江崎、武田、森本、中嶋、朴木、猪股、中島 |

### Phase 4: ベータ → MVPリリース

先行テスト後、展開。ここがスタート。

**タスク:** フィードバック収集、UI改善、AI精度チューニング、展開

**オーナー:** 未定 — チームで決める

## 横断的な役割（全フェーズを通して機能する）

| 役割 | 担当 | スコープ |
|------|------|---------|
| Product Owner | 江崎 | プロダクトの方向性・優先順位の最終判断 |
| Tech Lead | 江崎 | 技術的なことで迷ったときに最終判断を行う |
| Scrum Master | 川上 | スプリント運営・チームの障害除去 |
| 社内案件化推進 | 川上 | ステークホルダー調整・社内承認 |
| DevOps・CI/CD | 森本・中嶋・朴木・吉田 | 開発環境の整備・デプロイパイプライン構築・運用自動化 |

## チーム構成サマリ

| 役割 | 人数 | 担当 | 備考 |
|------|------|------|------|
| PO / TL | 1名 | 江崎 | プロダクトの方向性・技術的判断の最終意思決定 |
| SM | 1名 | 川上 | スプリント運営 + 社内案件化推進 |
| 企画リード | 1名 | 藤田 | Sprint 0 前半を主導。UX設計・プロトタイプ作成後、離脱 |
| 開発メンバー | 10名 | 中島, 絹田, まおまお, 森本, 中嶋, 朴木, 猪股, 武田, 小林, 吉田 | 経験年数3年未満の若手。各自が担当領域を持つ |

**合計: 13名**
