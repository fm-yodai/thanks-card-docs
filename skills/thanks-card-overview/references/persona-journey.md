# ペルソナ & ユーザージャーニーマップ

ありがとうカード プロジェクトのユーザー像と体験の流れを可視化する。3人のペルソナが描くエコシステムの全体像。

## 3人のペルソナとエコシステム

このプロダクトは、3つのタイプのユーザーと、プロダクト・開発チームが循環をつくることで成り立つ。

| ペルソナ | タイプ | 特徴 |
|----------|--------|------|
| 森山 隆志 | PRIMARY — 受動的 | 業務が多忙で業務以外への関心・優先度が低い。「なんのためにやるのか」に納得できないと動かない。面倒見がよく、若手に頼られたら助けてやりたい気持ちはある |
| 中西 康弘 | SECONDARY — 積極的 | 「面白そうじゃん」で動く。楽しいかどうかが基準。放っておいても使うが、楽しさが枯れると止まる。エコシステムの起点 |
| 河野 真一 | SUPPORTING — 無関心 | 業務以外のことに関心がない。人にも無関心。能動的には動かないが、存在自体がエコシステムの受け皿になる |

### 設計の大方針

全員がアクティブユーザーでなくてよい。積極的に使う中西さんと、たまに使う森山さんによって、プロダクトが継続的に使われ続ける状態を理想とする。河野さんは最も動かしにくいユーザー像として、設計の下限を示す。

- **森山さんがプライマリの理由：** このプロダクトが組織に根づくかは、森山さんのような「忙しくて業務以外に関心が薄い層」が使ってくれるかで決まる。設計上もっとも工夫が必要で、動いてくれれば組織全体に波及する層
- **中西さんがセカンダリの理由：** 設計の工夫がなくても使ってくれる層。ただし止まるとエコシステムの起点が弱まる重要な存在。プロダクトの継続的な進化で楽しさを枯らさない工夫が必要
- **河野さんがサポーティングの理由：** 受け取ってさえくれればよい。不快感・拒否感を生じさせないことを設計の下限とする

### エコシステムの循環

中西さん・開発チーム（若手）・管理職は互いにカードを送り合い、エコシステムを温める。その中から森山さんや河野さんにもカードが届く。森山さんは体験が積み重なれば返してくれるかもしれないが、強制はしない。河野さんは受け取るだけで十分。

- **送る側の役割分担：** 中西さんは楽しいから自然に送る。開発チーム（若手）は自らユーザーとして参加し最初の体験を意図的に作る。管理職が加わることで送る側の層が厚くなる。送る側が特定の誰かに依存して枯れない構造にすることが重要
- **前提条件：** エコシステムが回り続けるにはプロダクトの継続的な開発が必要であり、そのためのリソースと体制が維持されることが前提

### プロダクト進化のフィードバックループ

エコシステムが回り続けるには、ユーザーの行動データを収集・分析し、プロダクトを進化させ続ける仕組みが不可欠。

利用データ収集 → 分析・価値訴求（経営層へ定量報告）→ プロダクト進化（新機能・改善）→ 新しい楽しさ → ユーザーの継続利用 → ↻ループ

- **循環の条件：** 中西さんの楽しさを個人の性格や好奇心に依存させたままでは枯れる。ユーザーの行動データやフィードバックを開発チームが受け取りプロダクトを進化させ続けることで新しい楽しさが返ってくる。MVP時点から分析基盤（データ収集・蓄積）を組み込み、経営層への定量的な価値訴求を可能にしておくことが重要
- **前提条件：** 継続的な開発リソースの確保には、分析データに基づく経営層への価値訴求が不可欠

## ペルソナA：森山 隆志（もりやま たかし）

穏やかで面倒見がいい。心の中ではありがたいと思っているが、日々の業務に追われていてそれどころではない。

- **タイプ：** PRIMARY — 受動的タイプ
- **年齢：** 46歳
- **部署：** 開発部（バックエンド）
- **経験年数：** 20年以上（FREEMIND在籍12年）
- **家族構成：** 妻、中学生の息子、小学生の娘
- **性格：** 穏やか、職人気質、口下手だが面倒見はいい、現実的で感情より納得感で動く

### 普段の働き方
自社内で受託開発案件を担当。朝9時に出社して淡々とコードを書き、18時には退社して家族との時間を大切にしている。日中は案件の対応に追われていて、業務以外のことに時間を割く余裕がない。Chatworkは業務連絡中心に使い、雑談チャンネルでスタンプを押すくらいの参加はしているが、自分から雑談を投稿することはほぼない。

### 感謝に対するスタンス
後輩が助けてくれたとき、心の中では「ありがたいな」と思っている。対面で「ありがとう」と軽く言うか、Chatworkでスタンプを押す程度。感謝の気持ちがないわけではなく、わざわざ時間をかけて文章を書く余裕と動機がない。

### このペルソナを動かすカギ
森山さんの壁は「業務以外のことへの関心・優先度の低さ」。**「なんのためにやるのか」に納得できないと動かない。** 森山さんを動かすには3つの理由が必要。

1. **納得感** — 導入効果（コミュニケーション改善・心理的安全性向上・離職率低下など）を定量的・定性的に伝え、時間を使う価値があると理解してもらう
2. **面倒見のよさ** — 「若手に送ってあげて」という先輩としての気持ちに訴える
3. **受取体験** — カードをもらって嬉しかった経験の積み重ね。ただし嬉しいだけでは送る行動には至らない。①と②が前提

強制感・同調圧力・ネガティブな動機での参加誘導は逆効果。送る体験のハードルは極限まで下げる。

## ペルソナB：中西 康弘（なかにし やすひろ）

明るくて社交的。楽しいかどうかがすべての行動基準。

- **タイプ：** SECONDARY — 積極利用タイプ
- **年齢：** 49歳
- **部署：** 開発部（フロントエンド／チームリーダー）
- **経験年数：** 25年以上（FREEMINDが個人事業主の集まりだった時代から在籍、18年目）
- **家族構成：** 妻、高校生の娘
- **性格：** 明るい、社交的、好奇心旺盛、楽しいかどうかが行動基準

### 普段の働き方
技術が好きで、今でも手を動かすことに喜びを感じている。チーム内で自然とリーダー的な役割を引き受けるようになった。人と一緒に何かやるのが楽しいからやっている。Chatworkでは雑談チャンネルの常連。飲み会の幹事を自然とやるタイプ。

### 感謝に対するスタンス
普段から「ありがとう」「助かったよ」を口に出す人。感謝を伝えることに照れや抵抗がない。

### このペルソナを動かすカギ
中西さんは動かす必要がない。勝手に動く。問題は「動き続けてくれるかどうか」。楽しさを構造的に支えるのは「プロダクトの継続的な進化」。リアクション機能、お返し機能、植物の新しい成長段階——こういった進化が中西さんのモチベーションを維持するフィードバックループになる。中西さんが善意で「みんなも書こうよ」と働きかけたとき、森山さんのような人は引いてしまう点に注意。

## ペルソナC：河野 真一（こうの しんいち）

業務以外のことに関心がない。人にも無関心。悪意はなく、ただ興味がない。

- **タイプ：** SUPPORTING — 無関心タイプ
- **年齢：** 45歳
- **部署：** 開発部（バックエンド）
- **経験年数：** 22年（FREEMIND在籍13年目）
- **家族構成：** 独身、一人暮らし
- **性格：** 合理的、マイペース、業務以外のことに関心が薄い、人に対しても無関心

### 普段の働き方
長年の経験で自分のやり方が確立されている。効率を重視し、必要なことだけを最短距離でやるタイプ。Chatworkは業務連絡のみ確認していて、雑談チャンネルはほぼ見ない。社内イベントにも特に参加しない。

### 感謝に対するスタンス
感謝の気持ちがないわけではないが、「仕事をきちんとやるのは当然のこと」という感覚が根底にあり、いちいち感謝を形にすること自体がピンとこない。

### このペルソナを動かすカギ
動かそうとしなくてよい。重要なのは不快な思いをさせないこと。過剰な通知、利用を促すナッジ、催促リマインドは無関心を不快に変えてしまう。河野さんは「受け取る人」として存在するだけでもエコシステムの受け皿として機能している。

## 設計示唆まとめ

### UI / UX で必ず押さえること
- **Google認証はワンクリック。** メールのリンク → 即カード表示。摩擦があると森山さんも河野さんも離脱
- **セッション保持は長めに。** 河野さんの2回目訪問時にログインを求めない。最低でも数ヶ月は保持
- **カード送信は最短導線。** 宛先選択 → メッセージ入力 → 公開/非公開 → 送信。業務の隙間に30秒で完了
- **デフォルトは公開、非公開への切替を分かりやすく。** 森山さんは非公開を選ぶかもしれない
- **「私も送る」ボタンはカード表示画面の自然な位置に。** 圧力にはしない
- **コンピテンシーラベルはAIが裏でつける。** ユーザーに選ばせない
- **開封演出が受け取り体験を強化。** スキップ可能に
- **組織の木は全員で一つ。** 個人実績ではなく「みんなで育てている」感覚。送る頻度が低い人にプレッシャーを与えない
- **「送ってみませんか？」系のナッジは入れない。** 河野さんの無関心を不快に変え、森山さんも引く
- **個人の利用頻度を可視化しない。ランキングを作らない**
- **初見で迷わない導線。** 中西さんは直感的に使い始めたい

### 通知設計
- **カード受信時（メール）。** 全ペルソナ共通の唯一のトリガー
- **月1回の集計通知（メール）。** 森山さんには思い出すきっかけ。河野さんはスルー
- **木のレベルアップ通知（メール）。** 中西さんはChatworkで共有する話題にする
- **催促通知は絶対に入れない**
- **メール通知なのでストレスは低い**

### MVP後の最優先追加機能
- **リアクション機能** — 中西さんの送りっぱなし問題を解決し、森山さんの主な参加手段になる
- **投稿後の編集・公開/非公開変更** — 森山さんへの安心感を提供

### ローンチ戦略
- **導入効果を定量的・定性的に伝える** — 森山さんの納得感のための準備
- **「若手に送ってあげてください」の声かけ** — 森山さんの面倒見のよさに訴える
- **管理職を送る側に巻き込む** — 送る側の層を厚くする
- **開発チームが率先してカードを送る** — 初期のタイムラインを温める
- **森山さんを意図的に頼る＆巻き込む** — 開発中に技術的な相談やベータ版フィードバックをお願いし、愛着を作る

### リスクと対策
- **納得感を作れるか問題：** 導入効果を説得力のある形で提示できなければ、面倒見のよさも受取体験も機能しない
- **送る側が枯れる問題：** 中西さん・開発チーム・管理職が送る側の中心。いずれかが止まるとエコシステムが停滞
- **植物の成長上限問題：** 「ありがとうの森」（木が成長しきったら新しい木、過去の木は森として残る）を将来機能として検討
- **善意の圧力問題：** 中西さんの声かけが森山さんへの圧力にならないよう、利用頻度の非可視化・ランキング非作成で構造的に緩和
- **成功の定義を間違えない：** 全員がアクティブに送り合う状態は必須ではない。利用率だけをKPIにしない
- **エコシステム維持の条件：** プロダクトの継続的な進化が必要。そのためにはMVP時点から分析基盤を組み込み、経営層への定量的な価値訴求を可能にしておく
