# インセプションデッキ

ありがとうカード プロジェクトの全体像をチームで共有し、合意するための10枚のスライド。Sprint 0 の最初に実施。

## 01. 我々はなぜここにいるのか（WHY ARE WE HERE）

日々の承認と感謝を可視化し、エンゲージメントの向上を通じて組織の力を引き出す。

- 従業員エンゲージメントは、組織の生産性・定着率・収益性に直結する。その根幹にあるのが日常の「承認」——自分の行動が誰かに認められ、組織の価値観と結びつく実感だ。
- しかし日々の業務の忙しさの中で、感謝や承認は表面化されにくい。
- 多くの組織は行動指針やバリューを定義している。FREEMIND では5つのコアコンピテンシー（誠実、当事者意識、協働、挑戦への意欲、創造的思考）を策定した。しかしこうした行動指針は、**日常業務で意識されることはほとんどない**。「理想論」として受け止められている側面もある。
- **本プロジェクトでは、感謝を「記録される承認」に変え、行動指針の浸透とエンゲージメント向上の好循環を生み出す。**

**もう一つの目的:** 若手メンバーが0からプロダクトを作り切る経験を通じて成長すること。このプロジェクトは「プロダクト」と「人」の両方を育てる取り組み。

## 02. エレベーターピッチ（ELEVATOR PITCH）

> **従業員エンゲージメントを高めたい** 組織のための
> **「ありがとうカード」**というプロダクトは、
> **感謝 × AI ラベリングで承認を可視化する**アプリです。
>
> これは**感謝メッセージを送るだけで、AIが自動で行動指針（コンピテンシー）をラベル付け**でき、
> **チャットでのお礼や対面での「ありがとう」**とは違って、
> **承認が記録として蓄積され、行動指針が自然と共通言語になる**ことが特徴です。

### なぜ外部SaaSではなく自社開発なのか

- **若手の成長機会** — 若手は0からプロダクトを作り切る経験ができない。企画・設計・開発・リリース・運用の全工程を若手メンバーが体験する。
- **各組織の行動指針への最適化** — 組織固有の行動指針に完全に合わせたAIラベリングは、汎用SaaSでは実現できない。

## 03. パッケージデザイン（PRODUCT BOX）

もしこのプロダクトを一つの箱に入れて誰かに手渡すなら、どんな言葉を添えるか？

コンセプト: **忙しさの中で忘れかけた温もりを、そっと思い出させてくれるもの** — そんな存在を目指して。

### 選定コンセプト: 「種まき」（決定済み）

> **今日の「ありがとう」は明日の土壌になる。**
>
> 日々の「助かった」を言葉にする。ただそれだけのことが、気づけば組織の根を静かに強くしていきます。

特徴:
- 手のひらから届く
- 習慣が根を張る
- 空気が変わる

チーム議論の結果、「種まき」コンセプトを採用。日常の感謝が組織の土壌となり、根を張っていくというメタファーがプロダクトのビジョンに最もフィットするため。

## 04. やらないことリスト（NOT LIST）

スコープを守るために「やらないこと」を明確にする。境界線を引くことで、チームが迷わず進める。（暫定決定）

### やること（MVPに含める）

| 機能 | 詳細・備考 |
|------|-----------|
| ログイン | @freemind.co.jp のGoogleアカウントで認証。ユーザーマスタ自動同期はしない（手動管理） |
| 宛先選択 | 1名 / 複数名 / 未選択（任意）。複数名は1通が全員に届く。未選択＝全体向け投稿（公開必須） |
| カード送信 | ありがとうの内容をテキスト入力・送信 |
| AIバッジ自動付与 | 5つのコアコンピテンシーをAIが判定しラベリング。宛先未選択でも付与 |
| 送受信一覧 | 自分が送ったカード / 自分に送られたカードを一覧表示 |
| 全体タイムライン | 自分以外のカードも閲覧可能（公開カードのみ） |
| メール通知 | 未開封カードがあるとき + 月1集計通知。批判者層を刺激しない控えめな通知手段として選定 |
| 匿名送信 | 受信者には匿名、管理者は確認可能。匿名 × 宛先未選択の組み合わせは禁止（UIガード） |
| 公開 / 非公開 | 非公開＝全体タイムラインに非表示（送受信者のみ閲覧可）。宛先未選択は公開必須（UIガード） |
| ゲーミフィケーション | 組織の木（全員で一つの植物を育てる）+ ありがとうゲージ。個人の実績ではなく「みんなで育てている」感覚で、送る頻度が低い人にプレッシャーを与えない設計 |
| 自己分析 | コンピテンシー集計・自分の強み表示・ありがとうゲージ・月1集計通知 |
| 開封演出 | カード開封時にパッケージデザインに沿ったアニメーション |
| 送信時アニメーション | カード送信時にパッケージデザインに沿った演出 |
| 分析基盤 | 利用状況・KPIの分析用データを収集・蓄積する基盤。アプリ内管理画面は設けず、外部BIツールまたはParquetファイルで分析。経営層への定量的な価値訴求に使用 |

### 後でやること（MVPリリース後に順次追加）

- リアクション（多様な絵文字反応）
- チームごとのありがとう数の把握・視覚化
- 便乗（「私も！」投稿）
- 通知ON/OFF
- お返し機能
- リッチな検索
- AI文章生成
- 送信者への報酬
- AIラベリングカスタマイズ

### やらないこと（当面対応しない）

- **ランキング機能** — 競争を煽る仕組みは感謝の文化づくりの趣旨と合わない
- **人事評価との連動** — 気軽に使ってもらうことが最優先。評価と紐づくと身構えてしまう
- **スタンプ・イラスト送信 / テンプレート固定** — 自分の言葉で感謝を伝えることを重視
- **いいね / いいねランキング表示** — SNS的な「いいね」は人気投票化のリスクがある
- **通知カスタマイズ（時間帯制御 / ランダム通知）** — 通知の高度な制御は複雑度が高い。MVPはメール通知のみで十分
- **案件紐づけ / 多面評価の材料** — 業務管理・人事評価系の機能はスコープ外
- **アカウント自動同期（Google Workspace連携）** — 約60名規模なので手動管理で対応

## 06. 解決案を描く（SHOW THE SOLUTION）

### ユーザー体験フロー

ログイン → カード作成 → AIラベリング → 送信・通知 → タイムライン表示 → レポートで可視化

1. **ログイン** — Amazon Cognitoで認証
2. **カード作成** — 宛先を選び、感謝のメッセージを入力
3. **AIラベリング** — Amazon Bedrockがメッセージ内容から5つのコンピテンシーを自動判定し、タグをプレビュー表示
4. **送信・通知** — カードが保存され、相手に通知が届く
5. **タイムライン** — 全社のフィードに流れ、みんなの「ありがとう」が見える化
6. **レポート** — 月次で部署別のカード送受信数、コンピテンシー分布をグラフで可視化

### システム構成

全レイヤーをAWSサーバーレス構成で統一。月50 USDの予算制約の中で運用負荷とコストを最小化。

| レイヤー | 技術 | 説明 |
|---------|------|------|
| Frontend | Vue.js + TypeScript | カード送信・タイムライン・レポートの3画面構成。S3 + CloudFrontで配信 |
| Backend | Hono on AWS Lambda | API Gateway + Lambda上のHonoでREST APIを提供 |
| データ | DynamoDB | ユーザー・カード・部署マスタを管理。サーバーレスでスケーリング自動 |
| AI | Amazon Bedrock | 感謝メッセージから5つのコンピテンシーを自動判定。AWS内で完結 |
| 認証 | Amazon Cognito | 社員のログイン・セッション管理 |
| ホスティング | S3 + CloudFront | 静的ファイル配信。レスポンシブ対応（PC・スマホ両対応） |

### 制約条件

- **インフラ予算:** 月額 50 USD — サーバーレス構成でコスト最適化
- **利用規模:** 全社60名、4部署（営業・開発・DX企画・管理）
- **AWS統一:** 全レイヤーをAWS上に構築、運用負荷を最小化
- **レスポンシブ:** PC・スマートフォン両対応。MVPでは社内ネットワーク・PCのみだが、外販を見据えてモバイルファーストのデザインにしておく。

## 07. 夜も眠れない問題（WHAT KEEPS US UP AT NIGHT）

チームで洗い出した**16**のリスクを4カテゴリに分類し、対策を合意した。

### A — 業務優先度・時間確保（6件）

リスク:
- みんな業務が忙しくてコミットできない＆スクラムイベントに時間を食われ、肝心の作業が進まない
- 優先度が下がった結果、進捗が滞り、スコープを狭くしリリース
- 社内案件化されないことで限られた時間しかコミットできず、全然進まない
- 業務との兼ね合いで優先度が下がり、メンバーがオーナーシップを取れずに進まない
- 思ったより膨大な時間が若手勉強会に使われており、社内から制限を受ける
- 業務量の予測がつきづらく、その人の担当で作業が止まる

対策:
- 納期を遅らせ、品質と学びは優先的に確保する
- スプリント期間を1か月に変更（2週間分のタスク想定）、週2スクラムで開発時間を確保
- 段階的エスカレーション：スクラム → カンバン → ウォーターフォール
- 社内案件化を目指す。無理なら教育訓練を10%程度に抑え、目を付けられないように気を付ける

### B — チーム推進力・スクラム運営（2件）

リスク:
- ファシリテート or メンバの推進力不足でプロジェクトが空中分解する
- 手法が先行してスクラムイベントが形骸化する

対策:
- 稼働可能時間・ブロッカーの有無など客観指標を可視化する仕組みを導入
- 他メンバーがオーナーシップを持つ
- 形骸化時は段階的にカンバン → ウォーターフォールへ移行

### C — 社内訴求・プロダクト定着（6件）

リスク:
- 社内への訴求が不十分で、目的が伝わらないままリリースしてしまう
- 使い続けてもらうための設計が足りず、リリース直後だけ使われて定着しない
- プロダクトの効果を説明しきれず、開発が下火になる
- 導入時に「なんのためにやるのか」への納得感を作れず、受動層が動かない
- 積極利用者の善意の声かけ（「みんなも書こうよ」）が受動層への圧力になり、逆効果を生む
- 送る側（積極利用者・開発チーム・管理職）の層が薄くなり、カードが届かない人が出てエコシステムが停滞する

対策:
- 社内案件化のための価値訴求をしっかり組み立てる
- UX設計に明確に時間をかけ、エコシステムを設計する
- MVP時点から分析基盤（データ収集・蓄積）を組み込み、利用データに基づく経営層への定量的な価値訴求を可能にする
- 導入時に「なんのためにやるのか」を明確に伝え、受動層の納得感を作る。導入効果（コミュニケーション改善・心理的安全性向上等）を根拠として準備
- 管理職を初期から送る側に巻き込み、送る側の層を厚くする。送る側が特定の誰かに依存して枯れない構造にする
- 個人の利用頻度を可視化しない・ランキングを作らないことで、善意の声かけが圧力にならないよう構造的に緩和する

判断基準: Sprint 0終了時に社内案件化の目処が立たなければ教育訓練として縮小運営に切替。リリース後3ヶ月で利用状況を評価し継続判断。

### D — 技術・コスト（2件）

リスク:
- 生成AIやサーバーレスのコストが嵩み、経営層からアプリ停止を言われる
- やりたいように実装した結果、セキュリティ的に問題がある状態になる

対策:
- アーキテクチャ設計時にコスト試算をしっかり行い、AWS活用推進組織の学習用費用に按分する
- やりたいようにやらない。意思決定の理由を明確にする

## 08. 期間を見極める（SIZE IT UP）

約6ヶ月でMVPリリースを目指す。ただし硬い締切ではなく、学びを優先した目安として設定。

| フェーズ | 期間 | 内容 |
|---------|------|------|
| Sprint 0 | 1〜2ヶ月 | 準備期間：インセプションデッキ、UX設計、技術選定・環境構築、バックログ作成、DoD策定 |
| Sprint 1〜2 | — | コア機能開発：カード送受信 → AIラベリング → タイムライン表示 |
| Sprint 3〜5 | — | フィードバック & 拡充：UI改善、通知機能、AI精度チューニング |
| Sprint 6（目標） | — | ベータ → MVPリリース：先行テスト後、全社約60名へ展開 |

- **タイムラインの硬さ: 目安** — 6ヶ月は目安であり、硬い締切ではない。学びが最優先なので、必要に応じてスコープを調整して対応する。
- **スプリントの長さ: 1か月** — 「夜も眠れない問題」対策として当初の2週間から1か月に変更。2週間分のタスク量を想定し、残りの時間を開発作業に充てる。週2回のスクラムで進捗を確認。
- **開発手法の段階的切替** — 開発時間との兼ね合いを見て、スクラム → カンバン → ウォーターフォールと段階的に手法を変化させる方針。

## 09. トレードオフスライダー（TRADE-OFF SLIDERS）

すべてを最高にはできない。チームで優先順位を合意し、判断に迷ったときの羅針盤にする。以下はPOの初期案。

| 項目 | 優先度 |
|------|--------|
| 学び | MAX（最優先） |
| 品質 | 高め |
| 納期 | 中程度 |
| スコープ | 中程度（調整弁） |

- **学び = 最優先** — メンバーの成長機会がプロジェクトの最大目的。納期やスコープは多少犠牲にしても、学びの質を落とさない。
- **スコープ = 調整弁** — 遅延が発生したらスコープを削って対応。機能を減らしてでも、学びと品質を守る。

### 議論ポイント

- Q1. このスライダーの位置に違和感はある？特に品質と納期のバランスについて。
- Q2. 「学びのために納期を延ばす」はどこまで許容できる？稟議との兼ね合いは？
- Q3. 予算（50 USD/月）は固定制約？それとも交渉の余地がある？
