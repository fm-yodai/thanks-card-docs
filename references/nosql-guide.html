SECTION 01

NoSQLとは何か — RDBとの思考モデルの違い

NoSQLを学ぶ第一歩は、RDBとの設計思想の違いを理解することです。技術的な差異の前に、まず「どう考えてデータを扱うか」という思考モデルの違いを押さえましょう。

RDBの基本的な考え方（おさらい）

RDB（リレーショナルデータベース）は、皆さんが業務で使い慣れたデータベースです。改めて特徴を整理します。

RDB の思考モデル：「データをどう正規化するか」

テーブル設計：データを正規化し、重複を排除して整理する。ユーザー、カード、リアクションを別々のテーブルに分ける。

JOIN：必要なときにテーブルを結合してデータを取得する。複雑なクエリもSQLで柔軟に書ける。

ACID特性：トランザクションの原子性・一貫性・独立性・永続性を保証。データの整合性が非常に高い。

スキーマ：テーブル構造を事前に定義。すべてのレコードが同じカラム構成を持つ。

NoSQL の基本的な考え方

NoSQL の思考モデル：「データをどうアクセスするか」

非正規化：データの重複を許容し、アクセスに最適な形で保存する。読み取りが高速になる。

アクセスパターン駆動設計：「どんなクエリが来るか」を先に決め、それに合わせてデータ構造を設計する。

結果整合性：即座に全ノードが一致しなくても、最終的に整合する。その代わり高い可用性とスケーラビリティを得る。

水平スケーリング：サーバーを追加するだけで処理能力を向上できる。RDBの垂直スケーリング（サーバーの性能を上げる）とは対照的。

思考モデルの比較

🗃️

RDB の発想

「データをきれいに整理しよう」
→ 正規化してテーブルに分ける
→ 必要なときにJOINで結合
→ どんなクエリにも対応できる

⚡

NoSQL の発想

「データを速く取り出せるようにしよう」
→ アクセスパターンを先に列挙
→ パターンに最適化した構造で保存
→ 想定クエリが超高速になる

💡

大事なポイント：NoSQLはRDBの「代替」ではなく、異なるトレードオフの選択です。RDBは「柔軟なクエリ」を得る代わりに「スケーラビリティ」を犠牲にし、NoSQLは「スケーラビリティ」を得る代わりに「クエリの柔軟性」を犠牲にします。どちらが優れているかではなく、プロジェクトの要件に合う方を選びます。

SECTION 02

NoSQLの主要カテゴリ

NoSQLと一口に言っても、データの持ち方によって大きく4つのカテゴリに分かれます。それぞれ得意な用途が異なります。

CATEGORY 01

KVS（Key-Value Store）

代表：Redis, DynamoDB（シンプルモード）

最もシンプルなNoSQL。キーを指定して値を一発で取得する。辞書やハッシュマップのイメージ。読み書きが極めて高速。

用途：セッション管理、キャッシュ、設定値の保存

CATEGORY 02

ドキュメントDB

代表：MongoDB, Firestore

JSONライクなドキュメント単位でデータを管理。各ドキュメントが異なる構造を持てる（スキーマレス）。ドキュメント内の属性で柔軟にクエリ可能。

用途：Webアプリのデータ全般、CMS、カタログ

CATEGORY 03

ワイドカラムストア

代表：Cassandra, HBase

大量の列（カラム）を持つデータを効率的に格納。行ごとに異なるカラムを持てる。大規模な時系列データやIoTデータの処理に強い。

用途：時系列データ、IoT、ログ分析

CATEGORY 04

グラフDB

代表：Neo4j, Amazon Neptune

ノード（点）とエッジ（線）で関係性を表現。「AさんはBさんの友達で、BさんはCさんの上司」といった複雑な関係のたどりが得意。

用途：SNS、レコメンドエンジン、不正検知

💡

DynamoDBの位置付け：DynamoDBは「KVS + ドキュメントDBのハイブリッド」です。パーティションキー（PK）とソートキー（SK）でKVSとして高速にアクセスしつつ、各アイテムの属性はスキーマレスでドキュメントDB的に柔軟な構造を持てます。この「いいとこ取り」がDynamoDBの大きな特徴です。

SECTION 03

DynamoDBの基本概念

DynamoDBを使いこなすための基本概念を、身近な例え話を交えて解説します。

パーティションキー（PK）とソートキー（SK）

DynamoDBのテーブル設計の核となる概念です。図書館に例えると理解しやすくなります。

図書館に例えるPK/SK

パーティションキー（PK）= 図書館の棚番号
PKの値によって、データがどの「棚」（パーティション）に保存されるかが決まります。同じPKを持つデータは必ず同じパーティションに格納されます。

ソートキー（SK）= その棚の中の本の並び順
同じ棚（PK）の中で、SKの値によって順番に並びます。範囲検索（「2024年1月〜3月のカード」など）ができるのはこのおかげです。

PKだけのテーブル：棚番号だけで本を管理。1つの棚に1冊だけ。シンプルだが範囲検索ができない。

PK + SK の複合キーテーブル：棚番号 + 並び順で管理。1つの棚に複数の本を並べられ、順番で探せる。

PK/SK によるデータ配置のイメージ
PK: USER#001
（田中さん）
SK: CARD#2024-01-15お疲れ様カード
SK: CARD#2024-02-10ありがとうカード
SK: CARD#2024-03-05助かったカード
PK: USER#002
（鈴木さん）
SK: CARD#2024-01-20感謝カード
SK: CARD#2024-04-01応援カード
SK: ...追加可能

同じPKのアイテムは同じパーティションに格納され、SK順にソートされる

GSI（Global Secondary Index）

「別の角度から本を探すための索引」

元のテーブルは「ユーザーID（PK）→ 送信日時（SK）」で検索できるよう設計されています。しかし「部署ごとのカード一覧が見たい」となったらどうでしょう？ 元のキー構成では対応できません。

GSIの役割：元テーブルとは異なるPK/SKの組み合わせで検索できる「セカンダリインデックス」を作成します。

例：

• 元テーブル: PK=ユーザーID, SK=送信日時 → 「田中さんのカード一覧」を取得

• GSI-1: PK=部署, SK=送信日時 → 「開発部のカードを新しい順に」取得

• GSI-2: PK=受信者ID, SK=送信日時 → 「鈴木さんが受け取ったカード」を取得

※ GSIは最大20個まで作成可能。ただし増えるほどコストも増えるため、本当に必要なアクセスパターンに絞る。

キャパシティモード

💰

オンデマンドモード

使った分だけ課金。事前のキャパシティ設定が不要。トラフィックの予測が難しい場合や、小規模アプリに最適。

60名規模のありがとうカードプロジェクトではオンデマンド一択。

⚙️

プロビジョンドモード

事前にキャパシティを予約。安定したトラフィックがある場合にコスト最適化できる。Auto Scalingとの組み合わせも可能。

大規模サービスでトラフィックパターンが予測可能な場合向き。

WCU/RCU（キャパシティユニット）

読み書きの「通貨単位」

WCU（Write Capacity Unit）：1 WCU = 最大1KBのアイテムを1秒に1回書き込む能力。

RCU（Read Capacity Unit）：1 RCU = 最大4KBのアイテムを1秒に1回読み込む能力（強い整合性読み込み）。結果整合性読み込みなら0.5 RCUで済む。

オンデマンドモードでは：WCU/RCUは自動管理されるため、意識する必要はほぼありません。課金はリクエスト単位（100万リクエストあたり約$1.25）で計算されます。

🔑

最大のハードル：RDBの「まずテーブルを設計して、後からクエリを考える」から、「まずアクセスパターンを列挙してから、テーブルを設計する」への思考転換です。DynamoDBでは「このデータにどうアクセスするか？」が設計のスタート地点になります。

SECTION 04

テーブル設計パターン — シングルテーブル vs テーブル分割

DynamoDBのテーブル設計には大きく2つのアプローチがあります。どちらにもメリット・デメリットがあり、チームのスキルやプロジェクトの特性に応じて選択します。

シングルテーブル設計

全エンティティを1テーブルに格納するパターン

ユーザー、カード、リアクションなど、すべてのエンティティを1つのDynamoDBテーブルに入れます。PK/SKのプレフィクスでエンティティタイプを表現します。

例：
• PK=USER#001, SK=PROFILE → ユーザー情報
• PK=USER#001, SK=CARD#2024-01-15 → カード情報
• PK=CARD#abc, SK=REACTION#user002 → リアクション情報

✅

メリット

• トランザクション操作が容易（同一テーブル内）
• 1回のクエリで関連データをまとめて取得可能
• テーブル管理が1つで済む
• DynamoDBの性能を最大限に活かせる

⚠️

デメリット

• 学習コストが高い
• PK/SK設計が複雑になりがち
• データの可読性が低い（コンソールで見づらい）
• 設計ミスの影響範囲が大きい

テーブル・パー・エンティティ（マルチテーブル）設計

エンティティごとにテーブルを分けるパターン

RDB的な感覚で、エンティティごとに個別のテーブルを作成します。各テーブルのPK/SK設計がシンプルになります。

例：
• Users テーブル: PK=USER#001
• Cards テーブル: PK=CARD#abc, SK=送信日時
• Reactions テーブル: PK=CARD#abc, SK=USER#002

✅

メリット

• 直感的で学習コストが低い
• 各テーブルのPK/SK設計がシンプル
• データの可読性が高い
• RDB経験者が理解しやすい

⚠️

デメリット

• 複数テーブルにまたがるトランザクションが複雑化
• テーブル数が増える
• 関連データの一括取得には複数回クエリが必要
• テーブルごとのGSI管理が必要

比較表

シングルテーブル マルチテーブル
学習コスト 高い（PK/SKプレフィクス設計の理解が必要） 低い（RDB的な感覚で始められる）
データ可読性 低い（異なるエンティティが混在） 高い（テーブルごとに整理）
トランザクション 容易（同一テーブル内で完結） 複雑化の可能性（テーブルをまたぐ場合）
パフォーマンス 最適化しやすい（1回のクエリで関連取得） 複数クエリが必要になる場合がある
向いているケース アクセスパターンが明確、経験者がいる NoSQL初心者が多い、エンティティ間の関連が少ない
🔧

ElectroDB について：ElectroDB はどちらの方式もサポートしています。service 機能で複数テーブルをまとめて管理できるため、テーブル分割でも開発体験は損なわれにくいです。TypeScriptとの親和性も高く、型安全なデータアクセスが可能です。

💡

ありがとうカードプロジェクトでは：60名規模のアプリケーションでは、シングルテーブルのパフォーマンスメリットはほぼ出ません（データ量が少ないため）。チームの学習コストを優先してテーブル分割を選ぶのも合理的な判断です。

SECTION 05

DynamoDB vs RDS PostgreSQL — ありがとうカードでの比較

ありがとうカードプロジェクトの文脈で、DynamoDBとRDS PostgreSQLを多角的に比較します。

比較軸 DynamoDB RDS PostgreSQL
月額コスト ~$1（オンデマンド、60名規模） $15〜30（db.t4g.micro 最小構成）
運用負荷 フルマネージド（パッチ・バックアップ自動、サーバーレス） パッチ適用・ストレージ管理・インスタンス監視が必要
スケーラビリティ 自動水平スケール（テーブルサイズ・トラフィック無制限） 垂直スケール（インスタンスサイズ変更、ダウンタイムあり）
学習コスト 高い（PK/SK設計・アクセスパターン駆動の思考転換が必要） 低い（SQL・正規化はチーム全員が既知）
柔軟なクエリ 制約あり（事前設計したアクセスパターンのみ効率的） 高い（SQLで自由にJOIN・集計・サブクエリ）
分析（OLAP） Export to S3 + Athena で分析（ETLが必要） 直接SQLクエリで分析可能（ただしOLTPへの負荷に注意）
外販対応 マルチテナント（PKプレフィクスで分離） スキーマ分離 or DB分離
サーバーレス親和性 ネイティブ（HTTP API、コールドスタート影響なし） Aurora Serverless v2 対応可（コスト高め、接続管理が必要）
AI/エージェント親和性 構造がシンプルでAIが理解しやすい SQL生成でAIを活用しやすい（Text-to-SQL）
ジュニア学習価値 モダンクラウドDB経験（キャリアの幅を広げる） 業界標準SQLスキル強化（即戦力の深化）
変更容易性 属性追加・新エンティティ追加は容易（スキーマレス）。GSIもオンライン追加可能。ただしPK/SKの構造変更は既存データのマイグレーションが必要で最もコスト高 ALTER TABLE でカラム追加・型変更が可能。マイグレーションツール（Prisma Migrate 等）でスキーマ変更を管理。大規模テーブルのカラム追加はロックの考慮が必要
🔄

変更容易性の本質的な違い：RDBは「スキーマの変更は手間だが、クエリの変更は自由」。DynamoDBは「属性の追加は自由だが、アクセスパターンの変更は困難」。つまり、開発中に「このデータも取りたい」とクエリを追加したくなった場合、RDBならSQLを書くだけで済みますが、DynamoDBではGSI追加やテーブル設計の見直しが必要になることがあります。アクセスパターンがどの程度固まっているかが、DynamoDB採用の重要な判断基準です。

プロダクトが継続的に進化する場合の影響

ありがとうカードのようなプロダクトは、運用開始後に「こんなデータも見たい」「この機能も欲しい」と要件が膨らんでいきます。この進化に対して、DynamoDBとRDSで影響の出方が異なります。

進化シナリオ DynamoDB RDS PostgreSQL
新しい属性の追加
例: カードにタグ・公開範囲を追加		 スキーマレスなのでそのまま書き込むだけ。ただし新属性での検索にはGSI追加が必要 ALTER TABLE + CREATE INDEX で対応。60名規模ならロックも一瞬
エンティティ関係の複雑化
例: コメント・スレッド、OKR連携		 新エンティティのPK/SK設計が必要。関係が増えるほど設計の見直しが累積的に難しくなる 新テーブル + 外部キーで自然に拡張。JOINで関連データを一度に取得できるのがRDBの本領
分析要求の増加
例: 部署間交流分析、コンピテンシー偏り		 OLTP/OLAP分離で対応 — DB選定に依存しない（後述）
外販・マルチテナント化
例: 他社への提供、テナント分離		 PKプレフィクスで論理分離。テナント増加のコスト影響が極めて小さい（従量課金） スキーマ分離 or DB分離。テナント数に比例してコネクション管理・コストが複雑化
リアルタイム連携
例: カード受信通知、ダッシュボード更新		 DynamoDB Streams でデータ変更を即キャプチャ。Lambda連携がサーバーレスと自然に統合 LISTEN/NOTIFY や論理レプリケーションで対応可能。サーバーレス構成との相性はやや悪い
📊

OLTP/OLAP分離の考え方：「分析の柔軟性」はDB選定から切り離して設計できます。DynamoDB・RDSのどちらを採用しても、ETLでデータをS3（Parquet形式）にエクスポートし、Athena・Python・BIツール等で自由に分析する構成にすれば、OLTP側のDB制約に縛られません。つまり「DynamoDBだとアドホック分析ができない」という懸念は、OLAP層を分離することで解消できます。

OLTP/OLAP分離アーキテクチャ

アプリケーションが扱うリアルタイムの読み書き（OLTP）と、集計・分析（OLAP）を分離する設計パターンです。

[OLTP層] DynamoDB or RDS
  　↓　ETL（定期 or CDC）
[データレイク] S3（Parquet形式）
  　↓　クエリエンジン
[OLAP層] Athena / Python / BIツール

この構成なら、OLTP側のDBがDynamoDBでもRDSでも、Parquet化されたデータに対してSQLで自由にJOIN・集計・フィルタができます。「部署間カード交流分析」「コンピテンシー偏り分析」「送信頻度低下の検知」といった後出しの分析要求にも、OLTP側の設計変更なしで対応可能です。外販時も、テナントごとのParquetパーティションを分けることで分析基盤を共通化できます。

💡

この分離が意味すること：DynamoDBの最大の弱点とされる「柔軟なクエリが苦手」は、OLTP層の制約です。OLAP層を別途設計すれば、分析の柔軟性はDB選定に依存しなくなります。結果として、DB選定の判断基準は「OLTP用途での適合性」（コスト・運用負荷・サーバーレス親和性・スケーラビリティ）に絞り込めます。この観点では、DynamoDBの優位性がより明確になります。

💰

コスト差が大きい：RDS の最小構成でも月 $15〜30 かかり、50 USD 予算の30〜60%を占めます。DynamoDB なら月 $1 以下で済み、AI API やその他サービスに予算を回せます。限られた予算の中で、DB以外の機能（AI分析、通知機能など）にどれだけ投資できるかが変わってきます。

🎓

学習価値のトレードオフ：「普段の業務で使っているDB」がチーム全員 RDB（MySQL/PostgreSQL）なら、RDS を選ぶことで「業務直結スキルの強化」が得られます。一方、DynamoDB を選ぶと「普段触れないモダンDB体験」が得られます。どちらの学習価値を優先するかはチームの意向次第です。

SECTION 06

「いつNoSQLを選ぶべきか」判断フロー

プロジェクトのDB選定で迷ったときに使える判断フローです。上から順に質問に答えていくことで、RDBとNoSQLのどちらが適しているかの目安が得られます。

❶ アクセスパターンは事前に明確か？
No → RDB（柔軟なクエリが必要）
↓ Yes
❷ データ間のリレーションが複雑か？
Yes → RDB（JOINが必要）
↓ No
❸ 水平スケーリングが必要か？
Yes → NoSQL
↓ No でも...
❹ サーバーレス・フルマネージドを重視するか？
Yes → NoSQL（DynamoDB）
↓ No でも...
❺ 予算制約が厳しいか？
Yes → DynamoDB（無料枠 + 従量課金）
↓ No でも...
❻ チームにNoSQL経験者がいるか？
No → RDB or テーブル分割DynamoDB

ありがとうカードプロジェクトの場合

判断フローを当てはめると

❶ アクセスパターンは明確か → Yes
タイムライン表示、マイページ（送受信一覧）、カード詳細の3パターンが中心。事前に列挙しやすい。

❷ データリレーションは複雑か → No
ユーザー → カード → リアクションのシンプルな構造。複雑なJOINは不要。

❸ 水平スケーリングは必要か → 現時点では不要
60名規模ではスケーリングの心配はない。ただし外販を見据えるなら将来的に有利。

❹ サーバーレスを重視するか → Yes
AWS Lambda + API Gateway のサーバーレス構成を前提としている。

❺ 予算制約は厳しいか → Yes
月50 USDの制約の中で、DBコストは最小化したい。

❻ NoSQL経験者はいるか → 少ない
→ テーブル分割方式 + ElectroDB で学習コストを軽減。

✅

結論：ありがとうカードプロジェクトにとってDynamoDB は合理的な選択です。アクセスパターンが明確でリレーションがシンプル、予算制約が厳しく、サーバーレス構成との親和性も高い。DynamoDBの弱点とされる「柔軟な分析クエリ」は、ETLによるOLTP/OLAP分離（S3 Parquet + Athena等）で補完できるため、DB選定の障壁にはなりません。テーブル設計方式（シングルテーブル vs テーブル分割）は、チームのスキルレベルと学習目標に合わせてSprint 0 で議論して決定しましょう。
