DESIGN PRINCIPLES

設計原則

すべての構造的判断は以下の3原則から導出される。迷ったらここに立ち返る。

1

1操作 = 1ファイル

UseCaseもfeatureも、変更の単位を小さく閉じる。AIコーディングエージェントがスコープを特定しやすく、PRレビューも明快になる。

2

依存は上から下へだけ

route → UseCase → Domain → Repository（実装）の一方向。Repository interface はドメイン層に定義し依存性を逆転（DIP）。feature間も直接importしない。循環依存を構造で防ぐ。

3

契約は shared に集約

Zodスキーマ・エラー型・定数は packages/shared に。フロントとバックが同一の型から生成された定義を参照する。ズレが原理的に起きない。

💡

AIエージェント最適化：この構造は Claude Code / GitHub Copilot 等のAIコーディングエージェントが「このファイルを変更すれば良い」と即座に判断できることを重視している。1ファイル1責務、コンストラクタでの明示的な依存宣言、featureの公開API（index.ts）がエージェントのコンテキスト精度を高める。

各層の責務ルール

層責務禁止事項
routes/リクエスト受付・Zodバリデーション・UseCaseへの委譲・レスポンス整形ビジネスロジック、DB直接アクセス
handlers/SQS/Streams イベントのパース・UseCaseへの委譲ビジネスロジック、DB直接アクセス
use-cases/オーケストレーション。ドメインモデルの操作とRepository/Service呼び出しの調整ビジネスルール（状態遷移・バリデーション）の直接実装、HTTP関連の知識
domain/ビジネスルール・不変条件の保持。状態遷移メソッド・バリデーション・ドメインイベント定義。Repository/Service の interface を定義（依存性逆転）インフラ知識（DB固有API、HTTP）、外部ライブラリへの依存
repositories/domain/interfaces の DynamoDB 実装。データの永続化・取得ビジネスロジック、他Repositoryへの依存
services/domain/interfaces の外部API実装（Bedrock, SES）ビジネスロジック
di/全依存関係の組み立て。「具体実装を知る」唯一の場所ビジネスロジック
🔄

RDS移行パス：DynamoDB → RDS に切り替える場合、card.rds.repository.ts を追加し di/container.ts の注入先を差し替えるだけで完了する。Repository interface は domain/interfaces/ にあるため、UseCase層・ドメイン層は一切変更不要。

🧩

DI方式 — コンストラクタ注入：DIライブラリ（tsyringe等）は使用しない。UseCaseクラスのコンストラクタ引数でRepositoryとServiceを受け取り、di/container.ts で組み立てる。デコレータ不要でTypeScript設定がシンプル、AIエージェントが依存関係を即座に把握でき、テスト時はモックを差し込むだけ。

🏛️

ドメイン層（軽量ドメインモデル）：domain/models/ にはビジネスルールを持つドメインモデルを配置する。Card モデルは状態遷移メソッド（publish(), block(reason)）を持ち、不正な遷移をコンパイル時ではなく実行時に防止する。UseCase はドメインモデルのメソッドを呼ぶだけで、ビジネスルールを直接実装しない。これにより貧血ドメインモデル（Anemic Domain Model）を構造的に防ぐ。

🗺️

Bounded Context（論理境界）：MVPでは物理的なサービス分離は行わないが、use-cases/ 内をコンテキスト単位でサブディレクトリ分割する。カード送受信（Card, Timeline）、AIラベリング（LabelingResult, BlockReason）、ユーザー管理（User, Tenant）、通知（NotifyRecipient）の4境界。特にAIラベリングは独立性が高く、将来の別サービス切り出し候補。

feature間ルール

必須

直接importは禁止

card/ が timeline/ のファイルを直接参照しない。連携が必要な場合は index.ts の公開APIを通じて行う。

必須

index.ts で公開APIを明示

各featureは index.ts で外部に公開するコンポーネント・composable・型を明示する。内部のファイルは外から見えない。

必須

queries/ はfeature内に閉じる

TanStack Queryのquery key定義とhookをfeature内に持つ。キャッシュの管理単位が機能に紐づき、invalidateの制御が明確になる。

📦

状態管理：Pinia不使用。サーバー状態はTanStack Queryが管理。UIローカルの状態（モーダル開閉、フィルタ条件等）はcomposableの ref で十分。状態の所在がシンプルになり、デバッグしやすい。

🔗

Hono RPC による型安全API呼び出し：api/ ディレクトリで Hono の hc<AppType>() を使い、バックエンドのルート定義から型安全なクライアントを自動生成する。Zodスキーマ（packages/shared）→ Honoルート型 → RPCクライアント → TanStack Query の一気通貫で、APIの入出力型がコンパイル時に検証される。

🧪

テスト戦略：ユニットテスト（Vitest）+ E2Eテスト（Playwright）をMVPから導入。feature内の __tests__/ はVitest単体・composableテスト用。E2Eテストはプロジェクトルートの e2e/ ディレクトリに配置し、主要ユーザーフロー（カード送信・タイムライン閲覧）をカバーする。

SHARED PACKAGE

packages/shared 設計

フロントとバックの「契約」とドメイン共有型を定義する場所。Zodスキーマが全ての型の起点（Single Source of Truth）。contracts/ はAPI入出力の形、domain-types/ はドメイン概念の定義を担い、関心を明確に分離する。

ディレクトリ構造

🔍

AIブロックはエラー型に含めない。AIラベリングは非同期で実行され、ブロック判定はカードの状態遷移（status: 'blocked'）として扱う。ブロック理由付きのメール通知で送信者に伝える。フロントのエラーハンドリングとは完全に別フロー。

🔗

バックエンド → フロント連携：UseCaseが AppError をthrow → middlewareの error-handler.ts が code を見てHTTPステータスに変換 → フロント側は isAppError() type guardで判別 → code に応じたUI表示（Toastの色・メッセージを切り替え）。

DATABASE PACKAGE

packages/db 設計

ElectroDB を用いた DynamoDB エンティティ定義。Single Table Design のスキーマを型安全に管理し、apps/api の Repository 層から利用される。

🔑

📦

3層のスキーマ棲み分け：shared/contracts はAPI入出力の形、shared/domain-types はフロント・バック共通のドメイン概念（CardStatus, CompetencyId等）、db はバックエンド専用の永続化スキーマ。apps/api の domain/models/ はドメインの不変条件（状態遷移ルール等）を持つ。フロントは packages/db に依存せず、domain-types を通じてドメイン概念を共有する。

ASYNC LABELING

非同期ラベリング設計

送信即完了 → AIラベルは後から自然に表示。TanStack Queryのスマートポーリングで実現する（SSEは見送り）。

SSEを見送った理由

制約

Lambda 29秒制限

API Gateway + Lambdaの組み合わせではSSEの持続接続がLambdaタイムアウトに抵触する。

制約

コスト

WebSocket APIの追加は月$50の予算制約下でオーバーエンジニアリング。60名規模には不要。

代替

TanStack Queryで十分

スマートポーリング + window focus refetch で、SSEと遜色ないUXを実現できる。

3フェーズフロー

Phase 1: 送信（即時）

APIがDynamoDBに保存 → 即レスポンス。TanStack Queryのoptimistic updateでタイムラインにカードが即座に表示される。ラベル部分は「ラベリング中...」のシマー表示。

Phase 2: AI処理（非同期）

DynamoDB Streams → Worker Lambda → Bedrock → 結果をDynamoDBに書き戻し。ユーザーは待つ必要なし。画面を閉じてもOK。

Phase 3: フロント反映

TanStack Queryの2つのポーリング戦略がキャッチ。ラベルがCSSトランジションでふわっとフェードイン。

2つのポーリング戦略

戦略用途設定停止条件
A: post-send polling 送信直後に画面を見ている人向け refetchInterval: 5s（最大60秒） competenciesフィールドが埋まったら停止
B: background freshness 別画面から戻ってきた人向け refetchOnWindowFocus: true
staleTime: 30s		 常時有効（タイムライン全体に適用）
✨

UXディテール：ラベル未確定のカードはスケルトン/シマーで「何かが入る場所」を予約表示。ラベルが付いた瞬間、CSSトランジション（opacity 0→1, 300ms ease）でフェードイン。「忙しくてもストレスなく、何なら気持ちよく使える」体験を実現する。

📡

ドメインイベントによるモデリング：非同期フローはドメインイベント（CardSent, CardLabeled）として domain/events/ に明示的に定義する。UseCase が CardSent イベントを発行 → インフラ層（DynamoDB Streams）が配送 → Worker Lambda が CardLabeled を処理する構造。イベントの定義がドメイン層にあるため、インフラ（Streams → SQS → EventBridge）を差し替えてもドメインロジックは不変。テスト時は「イベントが発行されたか」を検証するだけで済む。
