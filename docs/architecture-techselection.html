🃏 サーバー状態

カードステータス表示 — ポーリング設計

⚠️

Lambda + API Gateway の制約：SSEは現実的でない。API Gateway HTTP API の接続時間上限は 29 秒。AIワークフローの完了まで数分かかる可能性があるため、SSEで「処理中→公開」をプッシュし続けることは構造的に不可能。WebSocket API を使えば可能だが、コネクション管理 Lambda・DynamoDB 接続テーブルが必要になりアーキテクチャが大きく複雑化する。

✅

採用：TanStack Query の refetchInterval によるステータス条件付きポーリング。カードの status === 'processing' の間だけ 8 秒間隔でポーリング。refetchInterval に関数を渡すことで、status が published または blocked に変わった時点でポーリングが自動停止する。カスタム composable を自作せず TanStack Query のネイティブ機能で完結するため、実装コスト・テスタビリティ共に優れている。

queries/useCardQuery.ts — TanStack Query によるポーリング実装export function useCardQuery(cardId: string) { return useQuery({ queryKey: ['cards', cardId], queryFn: () => api.cards[':id'].$get({ param: { id: cardId } }), // status が 'processing' の間だけ 8 秒ポーリング。 // published / blocked になると refetchInterval が false を返し自動停止する。 refetchInterval: (query) => query.state.data?.status === 'processing' ? 8000 : false, }) } // 利用側 — TimelinePage / CardModal / MyPage で共通利用 const { data: card } = useCardQuery(cardId)
💡

UX上の注意：カードモーダルを開いた際に status === 'processing' なら「コンピテンシー分析中…」のスピナーをバッジ欄に表示。タイムラインのカード一覧では処理中ラベルを薄いグレーで表示し、完了時に色付きバッジへ自然に切り替わる。TanStack Query のキャッシュにより、同じカードIDを複数箇所で参照しても API リクエストは重複しない。

Lambda 関数の粒度

採用：HTTPリクエスト用

🏗️ モノリシック Hono Lambda

全HTTPルートを1つのHonoアプリ・1つのLambda関数に集約。Hono RPCの型推論がシンプルに機能し、コールドスタートの管理箇所が1つになる。SST v3 の Api コンストラクタで定義。

採用：非同期ワーカー用

⚙️ 専用 Lambda（非同期処理）

SQSコンシューマ・EventBridgeトリガー・ETLは別Lambdaとして分離。HTTPリクエストとは完全に独立したスケーリングができ、タイムアウトも個別設定可能。

Step Functions の導入検討：現在のMVPフローは「フィルタ判定 → OK/NG分岐 → DB更新」とシンプルなため、aiWorker Lambda 内の if 分岐で十分対応可能。Step Functions を導入すると状態遷移の可視化・個別ステップのリトライ制御が得られるが、アーキテクチャの複雑性・コストが増加する。将来的に人間によるレビューステップの追加や複数AIモデルの並列実行が必要になった場合に導入を検討する。

ミドルウェアスタック

リクエストが内側に向かって通過する順序。

1 errorHandler() 未捕捉エラーを { error, message, requestId } 形式で JSON 化して返す（最外層で全体を捕捉）
↓
2 cors() 許可オリジンを app.example.com に限定（環境変数で切り替え）
↓
3 requestId() X-Request-ID ヘッダーを付与。ログの突き合わせに使用
↓
4 logger() JSON 構造化ログ { timestamp, event, userId, requestId, latencyMs }
↓
5 jwtAuth() Cognito の JWKS を使い JWT を検証。失敗 → 401。userId / isAdmin を ctx に格納
↓
6 zValidator() packages/shared の Zod スキーマでリクエストを検証。失敗 → 422
↓
7 route handler 実際のビジネスロジック。services/ 経由で DynamoDB / SQS / Bedrock にアクセス
🧅

オニオン構造：Hono のミドルウェアはオニオン（玉ねぎ）構造で動作する。errorHandler() を最初（最外層）に登録することで、内側の全ミドルウェア・ハンドラーで発生したエラーを捕捉できる。リクエストは外→内、レスポンスは内→外の順序で通過する。

🔐

認証フロー：Cognito Hosted UI + Authorization Code Flow with PKCE を使用。JWT検証は Hono ミドルウェアで JWKS を使い公開鍵検証（jose ライブラリ）。CORS は環境変数で許可オリジンを切り替え（dev: localhost, staging/prod: 各ドメイン）。

💬

フロントエンド認証SDKの検討：以下の候補から Sprint 0 で選定する。
① amazon-cognito-identity-js — Cognito専用の軽量ライブラリ。バンドルサイズが小さくシンプル。Cognitoにロックインされる。
② oidc-client-ts — 汎用OIDCライブラリ。将来的にCognito以外のIdPに切り替え可能で外販対応に有利。
③ @aws-amplify/auth — Amplify Auth単体利用。機能が豊富だがバンドルサイズが大きい（aws-amplifyパッケージ全体で数百KB）。ツリーシェイキングで軽量化可能。
外販への備え（方針3）を重視するなら②、シンプルさ重視なら①が有力。

🛡️

障害対策：DynamoDB Streams → aiWorker Lambda の構成により、DB書込が成功すれば非同期処理が確実にトリガーされる（SQS投入失敗による不整合リスクを排除）。aiWorker 側では DLQ（デッドレターキュー）を設定し、3回リトライ後に失敗レコードを退避。Bedrock API 呼び出し失敗時は最大3回再試行し、それでも失敗した場合は status を 'error' に更新してDLQに記録。CloudWatch Alarms でLambdaエラー率・DLQメッセージ数を監視し、Slackまたはメールで通知。

DynamoDB Streams 採用理由：従来の「DynamoDB書込 → SQS投入」フローでは、SQS投入失敗時に status: 'processing' のまま永久に処理されないカードが残るリスクがあった。DynamoDB Streams を使うことで、DB書込が成功すれば自動的に Lambda がトリガーされ、トランザクション整合性の問題を構造的に排除できる。Streams の処理失敗時は自動リトライ + DLQ で対応する。

DYNAMODB

DynamoDB シングルテーブル設計

全データを1テーブルに集約。PK/SKのプレフィクスでエンティティタイプを表現。DBクライアントは ElectroDB を使用し、PK/SKの組み立て・型推論・クエリビルドを宣言的に管理する。

⚙️

ElectroDB について：DynamoDB 専用の ORM 的ツール。エンティティ定義をスキーマとして宣言的に書くと、PK/SK の組み立て・型推論・クエリビルドを自動化してくれる。Single Table Design のベストプラクティスに沿った設計を自然に強制し、SST ドキュメントでも推奨されている。生の AWS SDK に比べてタイプミスによるキー設計崩壊を防ぎやすい。

マルチテナント対応：外販フェーズに備え、すべての PK に TENANT#<tenantId>| プレフィクスを付与するオプションを想定しておく。MVP 段階では固定値（例：TENANT#default|）を使うか省略し、外販フェーズで移行スクリプトで対応する。設計上の変更は最小限に抑えられる。

🔄

変更容易性：ElectroDB + シングルテーブル設計は以下の変更が容易 — 属性追加（スキーマレス）、新エンティティ追加（新PK/SKパターン）、GSI追加（オンライン追加可能）。一方、PK/SKの構造変更は既存データのマイグレーションが必要で最もコスト高。このため、リアクション等のMVP後機能もエンティティ設計だけは残しておき、後から追加できるようにしている。

RadarAgg — Write-time 集計の設計詳細

✅

Write-time aggregation を採用する理由：マイページを開くたびにカードを全件スキャンして集計（Read-time）は、将来データ量が増えるにつれてレイテンシが上昇する。一方、カードにコンピテンシーが付与された瞬間に RadarAgg アイテムを原子的加算（ADD）で更新しておけば、レーダーチャート表示は常に GetItem 1回で完了する。60名×12ヶ月の集計アイテムは合計 720 件程度で管理コストも低い。

packages/db/src/entities/radarAgg.ts — ElectroDB による RadarAgg 更新// ElectroDB エンティティ定義（packages/db 内で一元管理） export const RadarAggEntity = new Entity({ model: { entity: 'radarAgg', version: '1', service: 'thanksCard' }, attributes: { userId: { type: 'string' }, yearMonth: { type: 'string' }, // e.g. "2026-03" competencies: { type: 'map' }, // { ownership: 5, challenge: 3, ... } }, indexes: { primary: { pk: { field: 'PK', composite: ['userId'] }, sk: { field: 'SK', composite: ['yearMonth'] }, }, }, }, { table: TABLE_NAME, client: dynamoClient }) // aiWorker 内での更新（受信者ごとにインクリメント） for (const recipientId of card.recipientIds) { await RadarAggEntity.update({ userId: recipientId, yearMonth }) .add({ competencies: { [competency]: 1 } }) // 原子的加算 .go() } // GET /users/me/radar?month=2026-03 はこれだけで返せる const { data } = await RadarAggEntity.get({ userId, yearMonth }).go() // → { ownership:5, creative_thinking:2, challenge:3, integrity:1, collaboration:4 }
⚡

原子的加算の恩恵：DynamoDB の ADD 式は複数の aiWorker Lambda が同時に同じアイテムを更新しても競合しない。ElectroDB の .add() はこれをラップしており、ロック不要でスケールする設計になっている。アイテムが存在しない場合（初月）は ElectroDB が upsert として扱い、自動的に0から加算される。
