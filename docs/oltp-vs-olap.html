TECH FOUNDATIONS — 01

OLTP vs OLAP

「なぜ同じデータベースで全部やらないの？」——この疑問に答えるための基礎知識。 ありがとうカードのアーキテクチャで採用したOLTP/OLAP分離設計の背景を理解しよう。

🏗️

アーキテクチャ設計ノートとの関連

設計ノートの「分析アーキテクチャ」セクションでは、OLTP（アプリ内リアルタイム）とOLAP（バッチ集計）の二層構成を採用しています。この資料ではその判断の背景にある一般的な考え方を解説します。

CONCEPT

そもそもOLTPとOLAPって何？

どちらもデータベースの「使い方」のパターンです。同じデータを扱うけれど、目的がまったく違います。

OLTP — Online Transaction Processing

🔄 日々の業務処理

「カードを送る」「いいねする」「ログインする」など、ユーザーが1件ずつ行う操作を処理するためのデータベースの使い方。

特徴：1回のクエリで扱うデータは少量（1件〜数十件）。レスポンス速度が命。1秒以内に返ってこないとユーザーが待たされる。

OLAP — Online Analytical Processing

📊 分析・集計処理

「先月のカード送受信を部署別に集計」「コンピテンシーの分布を全社で可視化」など、大量のデータをまとめて分析する使い方。

特徴：1回のクエリで数千〜数万件を横断的にスキャン。処理に数秒〜数分かかっても許容される。

ありがとうカードでの具体例

OLTP：「田中さんがカードを送る」→ カードを1件INSERT → コンピテンシータグを紐づけ → 完了。数十ミリ秒で終わる。
OLAP：「先月、営業部から開発部に送られたカードのコンピテンシー分布は？」→ カードテーブルの全件スキャン → ユーザーの部署情報と結合 → コンピテンシー別にGROUP BY → ヒートマップ用のマトリクスを生成。

PROBLEM

なぜ1つのDBで両方やると問題になるの？

「60名規模だし、1つのDBで十分では？」——今は確かにそうです。でも、分離設計にする理由は規模だけではありません。

⚡ 負荷の衝突

OLAP の重い集計クエリ（全テーブルスキャン、複数JOIN）が走っている間、OLTP の軽い操作（カード送信）が遅くなる可能性があります。月次レポートのバッチ実行中にアプリが重くなる、というのは典型的なパターンです。

📐 データの持ち方が違う

OLTP は「行単位」でデータを読み書きします（カード1件のINSERT/SELECT）。一方、OLAP は「列単位」で大量データを集計するのが得意です。同じデータでも、最適な格納形式がまったく違います。

🔧 関心の分離

アプリ開発チームと分析チームでは、DBに求めるものが違います。アプリ側は「速いレスポンス」「安全なトランザクション」、分析側は「自由なクエリ」「大量データの高速スキャン」。責務を分けた方が、それぞれに最適化できます。

🌱 将来への備え

今は60名でも、外販を視野に入れた場合にユーザー数が拡大する可能性があります。最初からOLTPとOLAPを分離しておけば、規模が変わっても構造を変えずに対応できます。

💡

大事なポイント：60名規模の今、負荷の衝突は現実にはほぼ起きません。分離する一番の理由は「関心の分離」と「データの持ち方の違い」です。そして、分離設計そのものを構築する経験がメンバーの学びになるという点も大きいです。

ROW vs COLUMN

行指向 vs 列指向

OLTPとOLAPで「データの持ち方」が違う、とはどういうことかを具体的に見てみましょう。

例として、こんなテーブルがあるとします：

card_idsenderreceivercompetencymessage
001田中鈴木協働急な対応ありがとう！
002佐藤田中誠実丁寧なレビューに感謝
003山田佐藤挑戦への意欲新しい提案、素晴らしい

📋 行指向ストレージ（RDB / DynamoDB）

データを行単位でまとめて保存します。
※ DynamoDB は厳密にはKey-Value/ドキュメントストアですが、「1アイテム（行）を丸ごと読み書きする」点で行指向と同じ特性を持ちます。

// ディスク上のイメージ [001, 田中, 鈴木, 協働, 急な対応…] [002, 佐藤, 田中, 誠実, 丁寧な…] [003, 山田, 佐藤, 挑戦, 新しい…]

「card_id=001 のカードを全部取得」が高速。OLTPに最適。

📊 列指向ストレージ（Parquet）

データを列単位でまとめて保存します。

// ディスク上のイメージ card_id: [001, 002, 003] sender: [田中, 佐藤, 山田] receiver: [鈴木, 田中, 佐藤] competency: [協働, 誠実, 挑戦] message: [急な対応…, 丁寧な…, 新しい…]

「competency列だけ全件読み取ってGROUP BY」が高速。OLAPに最適。

📦

Parquet とは：Apache Parquet はオープンソースの列指向ファイルフォーマットです。データを列ごとに圧縮して保存するため、「特定の列だけ読む」集計クエリが非常に高速。AWS Athena、Python Polars/Pandas など多くのツールが対応しています。今回のETLパイプラインでは、DynamoDB/RDSからデータを抽出してParquet形式でS3に書き出します。

ETL PIPELINE

ETLパイプラインという橋渡し

OLTP と OLAP を分離するとき、「OLTP側のデータをOLAP側に持っていく仕組み」が必要です。これがETLパイプラインです。

ETL方式の候補（Sprint 0 時点で未定）：
① DynamoDB Export to S3 — ネイティブ機能でLambda不要。60名規模のMVPでは最もシンプルで推奨。
② Lambda ETL — EventBridge → Python Lambda → Scan → Parquet変換。柔軟だが実装コストあり。
③ DynamoDB Streams — CDC方式でニアリアルタイム。MVP段階ではオーバースペック。

Transform の具体例 — 部署間ヒートマップ

カードテーブルには sender_id と receiver_id しかありません。部署情報はユーザーテーブルにあります。
Transform では、カードデータにユーザーの部署情報を結合（JOIN）し、「営業部→開発部: 12件、開発部→DX企画部: 5件…」というマトリクスを生成します。
この結果をParquetに書き出しておけば、QuickSightやPythonスクリプトで即座にヒートマップを描画できます。

TECH STACK

今回使う技術

OLAP/ETL 周辺で登場する技術群の位置づけを整理します。

技術役割なぜ選んだか
Amazon S3 Parquetファイルの保存先 ほぼ無制限の容量。コストは月数セントレベル（この規模では）
Apache Parquet 列指向ファイルフォーマット 列指向で集計に最適。圧縮率が高い。業界標準のオープンフォーマット
AWS Athena S3上のParquetにSQLを投げられるサービス サーバー不要。スキャンしたデータ量だけの従量課金（$5/TB）
Python（Polars / Pandas） ETL処理の実装言語・ライブラリ Lambda上でのデータ変換に使用。具体的なライブラリは検討中（Polarsは高速・省メモリ、Pandasはエコシステムが豊富）
Amazon QuickSight BIダッシュボード S3/Athenaと直接接続可能。定型レポートの可視化に使う（将来の外販向け）
🎓

学びの観点：ETLパイプラインの設計・構築はデータエンジニアリングの基礎です。Parquet、Athena、Polars といった技術群に実データで触れる経験は、データ分析や機械学習を志向するメンバーにとって貴重な学びになります。
