デザインシステム入門

「デザインシステム」ってなに？を理解するためのガイド

このプロジェクトではデザインシステムを使って、アプリの見た目と使い心地の一貫性を保ちます。「デザインシステム？聞いたことはあるけど、よくわからない」という方向けに、基本的な考え方をまとめました。

1デザインシステムとは

デザインシステムとは、アプリの見た目や使い方のルールを体系的にまとめたものです。色やフォントなどの基本的な値から、ボタンやカードなどのUI部品、それらの使い方のルールまでを含みます。

レシピブックのたとえ

料理に例えると、デザインシステムは「レシピブック + 調味料棚」のようなものです。

  • 調味料棚（トークン）: 醤油、みりん、塩… 使える材料が決まっている
  • 基本レシピ（コンポーネント）: 出汁の取り方、ご飯の炊き方… よく使う手順がまとまっている
  • 献立の組み方（パターン）: 主菜＋副菜＋汁物… 組み合わせのルールがある
  • お店の方針（ガイドライン）: 「素材の味を活かす」「化学調味料は使わない」… 判断の基準がある

誰が作っても同じ味になる。それがデザインシステムの目指すところです。

デザインシステムは通常、以下の要素で構成されます：

  • デザイントークン — 色、余白、フォントサイズなどの「値」
  • コンポーネント — ボタン、カード、フォームなどの「部品」
  • パターン — 部品をどう組み合わせるかの「型」
  • ガイドライン — 文言のトーンや操作感の「ルール」

2なぜ必要なのか

60人が使う社内アプリをチームで開発するとき、ルールなしだとどうなるでしょう？

「このボタンは青だけど、あっちの画面では緑になってる」「同じ機能なのにページごとに操作感が違う」——こうした問題を防ぎ、チーム全員が同じ基準で開発できるようにするのがデザインシステムの役割です。

🎯

一貫性

どの画面を開いても、同じ色・同じボタン・同じ操作感。ユーザーが迷わないアプリになります。

🚀

開発スピード

毎回ゼロからUIを考えなくていい。既存の部品を組み合わせるだけで画面が作れます。

🗣️

共通言語

「そこはPrimaryボタンで」「CompetencyBadgeを使って」——名前が揃うとコミュニケーションがスムーズに。

🤖

AI精度の向上

AIコーディングエージェントは、明確なルールがあるほど正確なコードを書きます。デザインシステムはAIへの「指示書」でもあります。

このプロジェクトならではのポイント 私たちはClaude CodeやGitHub Copilotなど、AIエージェントを活用して開発します。デザインシステムを整備することで、AIが生成するコードの品質が大幅に向上します。「AIルールファイルの品質がすべてのAI生成コードの品質を決定する」——これはデザインシステムの保守ルールで定められている大事な考え方です。

3構成要素を知る

デザインシステムは複数の層（レイヤー）で構成されます。下の層ほど基礎的で、上の層ほどプロジェクト固有になります。

L4 ガイドライン 使い方の「約束」
L3 ドメインコンポーネント プロジェクト固有のUI部品
L2 基盤コンポーネント 汎用UI部品
L1 デザイントークン すべての土台となる「値」

L1 デザイントークン

デザインの最小単位です。色、余白、フォントサイズ、角丸、影、アニメーション速度などの「値」をCSS変数として定義します。

ポイントは、コードの中で#FF6B6Bのような具体的な値を直接書かないこと。代わりにvar(--ds-color-primary-500)のようなトークンを使います。こうすることで、色を変更したくなったときにトークンの値を1箇所変えるだけで、アプリ全体に反映されます。

L2 基盤コンポーネント

ボタン、テキスト入力、バッジ、モーダルなど、どんなプロジェクトでも使える汎用的なUI部品です。これらはトークンだけを参照し、プロジェクト固有の知識を持ちません。

L3 ドメインコンポーネント

「感謝カード（ThankYouCard）」や「コンピテンシーバッジ（CompetencyBadge）」など、このプロジェクト固有のUI部品です。L2の部品を組み合わせて作ります。

L4 ガイドライン

レイアウトの組み方、エラー表示のルール、文言のトーンなど、「部品をどう使うか」の約束事です。

4日常の開発でどう使うか

デザインシステムは作って終わりではなく、日常の開発の中で使い続けることで力を発揮します。立場ごとの関わり方を見てみましょう。

Developer

開発メンバー

UIを作るときは、まず既存のコンポーネントを探す。色や余白はトークンを使う。新しい部品が必要なら、チームに相談。

「ボタンが必要？→ Buttonコンポーネントを使おう。色は？→ var(--ds-color-interactive-default) だね」
Reviewer

レビューする人

PRレビューで、ハードコードされた色値がないか、既存コンポーネントで代替できないか、命名規約に沿っているかをチェック。

「この #FF6B6B はトークンに置き換えられそう。あと、このUI部品は既存のBadgeで作れない？」
AI Agent

AIエージェント

デザインシステムのドキュメントを自動的に読み込み、ルールに従ったコードを生成。人間がルールを覚えていなくてもAIが守ってくれる。

「tokens.cssのトークンを使用し、既存のStackコンポーネントでレイアウトを構成しました」
スプリントごとの健全性チェック スプリントごとの振り返りで、10〜15分のデザインシステム健全性チェックを行います。ハードコード値が混入していないか、新しい共通パーツがコンポーネント化されているかなどを確認します。

5このプロジェクトのデザインシステム

ありがとうカードプロジェクトのデザインシステムは、「種まき」というコンセプトを世界観の起点としています。

「忙しさの中で忘れかけた温もりを、そっと思い出させてくれるもの」——この想いが、温かな色調、控えめなアニメーション、押しつけないUIのトーンに表れています。

デザインシステムの詳しい仕様（トークン定義、コンポーネント一覧、命名規約、保守ルールなど）は、デザインガイドラインにまとまっています。
