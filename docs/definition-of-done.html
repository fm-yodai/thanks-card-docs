Definition of Done（DoD）
ありがとうカード プロジェクト

プロダクトバックログアイテムが「完成」と見なされるための品質基準

DRAFT — チーム議論用叩き台 対象: 開発チーム全員 Sprint 0
📝

この資料は「叩き台」です

以下のDoD案は、プロジェクトの状況（12名チーム・Scrum・1か月スプリント・AWSサーバーレス・予算50USD/月）を踏まえた初期提案です。チーム全員で議論し、合意の上で確定させてください。項目の追加・削除・表現の修正は大歓迎です。DoDの詳しい背景知識については「DoD入門ガイド」を参照してください。

SECTION 01

DoDの方針

ありがとうカードプロジェクトのDoDを策定するにあたり、以下の方針を提案します。

🎯

小さく始めて育てる

初期DoDは確実に守れる必須項目（6項目）だけに絞ります。推奨・将来候補は分けて記載し、チームの開発リズムが安定した後、レトロスペクティブで段階的に昇格させていきます。

🤖

自動化できるものは自動化

CI/CDで自動チェックできる項目（型チェック・Lint・テスト・ビルド）はGitHub Actionsに集約し、人力チェックの負荷を最小限にします。CIグリーンを1項目として束ねることで、開発者が本質的な品質に集中できます。

プロジェクトの前提条件

チーム構成：12名（PO/TL 1名・SM 1名・開発メンバー 10名） — インセプションデッキ参照

開発手法：スクラム（1か月スプリント） — 開発の進め方ガイド参照

技術スタック：TypeScript / Vue 3 / Hono / DynamoDB / SST v3 / GitHub Actions — 統合設計ノート参照

品質ツール：Oxlint + ESLint（eslint-plugin-vue）+ Prettier / Vitest

対象ユーザー：60名（4部署）の社内利用

予算：月50 USD（インフラ + AI API）

📐

3段階の分類：このDoDでは項目を「必須」「推奨」「将来」の3段階に分けています。初期DoDとして運用するのは「必須」の6項目のみです。「推奨」はできれば満たしたい項目、「将来」はチームの成熟後に必須への昇格を検討する項目です。この分類自体もチームで議論して調整してください。

SECTION 02

必須 — 毎回必ず満たす（6項目）

以下のすべてを満たしたとき、PBIは「Done」とします。自動チェック可能な項目はCI/CDに集約し、人力で確認すべき項目を明確に分けています。

MUST — 初期DoDとして運用
  • 1. コードレビューで1名以上の承認（Approve）を得ている なぜ: コードの可読性・設計妥当性の確認と、チーム内の知識共有を兼ねる。12名チームでは担当領域が分散するため、レビューを通じて他領域のコードを理解する機会としても機能する。GitHub ブランチ保護ルールで強制する
  • 2. CIパイプラインがすべてグリーンである なぜ: 以下の自動チェックを一括で担保する項目。手動で個別確認する負荷を排除し、マージ条件として GitHub ブランチ保護ルールで強制する
    　・TypeScript 型チェック（tsc --noEmit）
    　・Oxlint + ESLint（eslint-plugin-vue）
    　・Prettier フォーマットチェック
    　・Vitest 単体テスト実行（既存テストのリグレッション検知を含む）
    　・ビルド確認（Vite ビルド + SST synth）
  • 3. 新規ビジネスロジックに対する単体テスト（Vitest）が追加されている なぜ: コンピテンシー自動判定・バッジ付与条件・ストリーク計算など、コアロジックの正確性を担保する。UIコンポーネントの単体テストは必須としない。CIが既存テストのパスを保証するため、「既存テストのリグレッションなし」は本項目の自動チェックに含まれる
  • 4. 主要なユーザーフローを手動で動作確認済み なぜ: 自動テストだけでは拾えないUI/UXの問題（表示崩れ・操作感の違和感・遷移の不自然さ）を目視で確認する。変更した機能に関連するフロー（例: カード送信を変更したら、送信→AI判定→タイムライン表示の一連を確認）を対象とする
  • 5. デザインガイドラインに準拠している なぜ: 12名が並行開発するとUIの一貫性が崩れやすい。チームで策定済みのデザインガイドライン（カラー・タイポグラフィ・コンポーネント）に沿っているかをコードレビュー時に確認する。shadcn-vue + Tailwind CSS の共通コンポーネントを使うことで準拠の判断をしやすくする
  • 6. プロダクトオーナーが受け入れ基準（AC）を確認し、承認している なぜ: 作ったものがPO（江崎）の意図する要件と合っていることを確認する。「作ったけど求めていたものと違った」を防ぐ最終ゲート。確認タイミングはマージ前を推奨するが、チームで議論して決定する

議論ポイント: 必須項目

  • コードレビューの承認者数は1名で十分か？ 12名チームでは特定領域（認証・AI判定など）に詳しい人を指名レビュアーにすべきか？
  • POの確認タイミングはいつが最適か？（マージ前？ スプリントレビュー時？ 非同期で？）
  • 「デザインガイドライン準拠」の判断はレビュアー全員がするのか、デザイン担当（江崎）が確認するのか？
  • 手動テストの範囲をどう定義するか？（変更した機能のみ？ 関連フローも含む？）
SECTION 03

推奨 — できれば満たす（4項目）

チームの開発リズムが安定したら、レトロスペクティブで「必須」への昇格を検討する項目です。Sprint 1 の時点では「できれば」で運用し、負荷を見ながら判断します。

RECOMMENDED — 習熟後に必須へ昇格を検討
  • 開発環境（SSTステージング）にデプロイして動作確認済み なぜ: ローカル環境とAWSクラウド環境の差異（Lambda実行環境・DynamoDB接続・Cognito認証フロー）による問題を事前に検出する。SST v3 のステージ機能で環境分離が可能（統合設計ノート参照）。ステージング環境のコスト・運用体制が整い次第、必須への昇格を検討する
  • APIの追加・変更がある場合、API仕様が更新されている なぜ: Hono RPC（hono/client）で型安全なAPI呼び出しが可能なため、フロント・バックエンド間の型ズレはコンパイル時に検知できる。ただし、APIの振る舞い（認可ルール・バリデーション・レスポンス構造の意図）はコードだけでは伝わらないため、ドキュメント化しておくとチーム内の認識合わせに有効
  • 環境変数・設定値の変更がある場合、READMEまたは設定ドキュメントが更新されている なぜ: 12名が並行開発する中で、ローカルセットアップやデプロイ時に「動かない」を防ぐ。SST のリンク機能で環境変数は管理されるが、新規追加時の設定手順はドキュメントで共有する必要がある
  • モバイル表示（スマートフォン幅）で表示崩れがない なぜ: インセプションデッキでレスポンシブ対応は確定要件。外販に備え、スマートフォンからアクセスするケースを想定する。Tailwind CSS のレスポンシブユーティリティで対応しやすい環境は整っている

議論ポイント: 推奨項目

  • この中で最初から「必須」に入れるべきものはあるか？ 逆に「将来」に移すべきものは？
  • ステージング環境は初期から用意するか？ 予算（50USD/月）内で賄えるか？
  • Hono RPC の型共有があればAPI仕様書は不要という判断もあるが、チームの合意は？
SECTION 04

将来 — チーム成熟後に追加を検討（4項目）

Sprint 1 時点では含めないが、プロジェクトの進展やチームの成長に応じて「推奨」→「必須」へ段階的に昇格させる候補です。

FUTURE — Sprint 2以降で段階的に追加
🧪

E2Eテスト

主要ユーザーフロー（カード送信→AI判定→タイムライン表示）のリグレッションを自動検知する。E2Eツールは MVP 後半で選定予定（統合設計ノート参照）。

追加時期の目安: Sprint 2〜3（基本機能が安定した後）

♿

アクセシビリティチェック

axe-core等のツールで自動チェック。キーボード操作、スクリーンリーダー対応、コントラスト比の確保。

追加時期の目安: Sprint 3〜4（UI実装が進んだ後）

🔒

セキュリティチェック

依存パッケージの脆弱性スキャン（npm audit / Dependabot）。Cognito認証バイパス・XSSの確認。統合設計ノートのセキュリティ設計（v-html禁止・Zod入力検証・IAM最小権限）を項目化する。

追加時期の目安: Sprint 2〜（認証機能実装時）

⚡

パフォーマンス基準

主要ページの初回表示が3秒以内。Lighthouse スコアが一定以上（例: Performance 80+）。Lambda コールドスタートの影響も含めた計測。

追加時期の目安: Sprint 4〜（機能が出揃った後）

📅

レトロスペクティブのたびに見直す：「今のDoDで困っていることはないか」「推奨・将来から必須に昇格すべき項目はないか」をスプリントレトロスペクティブで定期的に振り返りましょう。チームの成長に合わせてDoDも進化させることが大切です。

SECTION 05

DoDの運用方法

DoDを策定した後、チームの日常にどう組み込むかを提案します。

PRテンプレートに組み込む

GitHub のプルリクエストテンプレートにDoDのチェックリストを埋め込みます。PRを作るたびに自動でリストが表示されるため、チェック漏れを構造的に防止できます。「必須」のみをテンプレートに含め、「推奨」は任意チェック欄として分離します。

## Definition of Done（必須） - [ ] コードレビューで1名以上のApproveを得た - [ ] CIがすべてグリーン（型チェック・Lint・テスト・ビルド） - [ ] 新規ビジネスロジックに Vitest 単体テストを追加した - [ ] 主要フローを手動で動作確認した - [ ] デザインガイドラインに準拠している - [ ] POが受け入れ基準を確認・承認した ## 推奨（該当する場合） - [ ] ステージング環境で動作確認済み - [ ] API仕様/ドキュメントを更新した - [ ] モバイル表示を確認した

自動化する項目

以下の項目はCIパイプライン（GitHub Actions + SST）で自動チェックし、人力チェックの負荷を軽減します。

GitHub Actions で自動実行：
・TypeScript 型チェック（tsc --noEmit）
・Oxlint + ESLint（eslint-plugin-vue）チェック
・Prettier フォーマットチェック
・Vitest 単体テスト実行
・Vite ビルド確認

GitHub ブランチ保護ルール：
・PRに1名以上のApproveを必須化
・CIチェックのパスをマージ条件にする
・mainブランチへの直接プッシュを禁止

※ ブランチ命名規約（feature/ / fix/ / chore/ / docs/）の検証もGitHub Actionsで実施予定（統合設計ノート参照）

スプリントイベントでの活用

スプリントプランニング：PBIの見積もり時にDoDの工数も考慮する。「テストを書く時間」「レビュー対応の時間」を含めてストーリーポイントを付ける。

デイリースクラム：「DoDのここでブロックされている」を共有し、チームで解決を図る。12名チームではレビュー待ちがボトルネックになりやすいため、レビュー依頼の優先度共有が重要。

スプリントレビュー：DoDを満たしていないPBIはデモに含めない。「Done」のPBIだけをインクリメントとして扱う。

レトロスペクティブ：DoDの運用に問題がないか振り返る。形骸化している項目はないか、「推奨」から「必須」に昇格すべき項目はないかを議論する。
