アーキテクチャ設計ノートとの関連

BASICS

認証と認可は別の概念

英語だと Authentication と Authorization。似ているけれど、まったく違う概念です。

AUTHENTICATION — 認証

🔑 「あなたは誰？」

ユーザーが本人であることを確認するプロセス。
Googleアカウントでログインする、パスワードを入力する、指紋で認証する——これらはすべて「認証」です。

例：田中さんが @freemind.co.jp のGoogleアカウントでログイン → 「あなたは田中さんですね」と確認が取れる。

AUTHORIZATION — 認可

🛡️ 「何をしていい？」

認証されたユーザーに対して、「何を許可するか」を判定するプロセス。
管理者だけがレポートを見られる、自分が送ったカードだけ削除できる——これらは「認可」です。

例：田中さんはログイン済み（認証OK）。でも管理者ではないので、月次レポートの管理画面にはアクセスできない（認可NG）。

ありがとうカードでの認証・認可

認証：@freemind.co.jp のGoogleアカウントでログインできること。これは Cognito + Google OAuth が担当。
認可：一般ユーザーはカード送受信とマイページの閲覧のみ。管理者は分析ダッシュボードにもアクセス可能。これはバックエンドの Lambda で制御します。

OAUTH 2.0

OAuth 2.0 ってなに？

「Googleでログイン」ボタンの裏側で動いている仕組みです。パスワードを直接扱わず、安全にログインを実現するための業界標準プロトコルです。

🤔

なぜ自前でパスワード管理しないの？ — パスワードの保存は非常にリスクが高い（漏洩したら全ユーザーに被害が及ぶ）。Googleのような信頼できるサービスに認証を委託すれば、パスワード管理の責任を負わずに済みます。ユーザーも新しいパスワードを覚える必要がなく、使い慣れたGoogleアカウントでログインできます。

OAuthの核心は 「パスワードを渡さずに、権限だけを渡す」 という考え方です。

ユーザー本人。
「自分のGoogleアカウント情報を、ありがとうカードに渡していいよ」と許可する人。

📱 クライアント

ありがとうカードのアプリ。
ユーザーの代わりにGoogleからプロフィール情報を取得したいアプリ。

🔐 認可サーバー

Google（と Cognito）。
「この人は本人です」「この情報を渡してOKです」を判定するサーバー。

AUTH FLOW

Authorization Code Flow + PKCE

ありがとうカードで使う認証フローです。OAuth 2.0 の Authorization Code Flow に PKCE（Proof Key for Code Exchange）を組み合わせたパターンで、SPAのような「クライアントシークレットを安全に保持できない」環境で推奨されます。

1

ユーザーが「Googleでログイン」を押す

ありがとうカードのログイン画面で、ユーザーがボタンをクリック。アプリはランダムな code_verifier を生成し、そのハッシュ値（code_challenge）を添えて Cognito の認証エンドポイントにリダイレクトします。

2

Cognito → Google にリダイレクト

Cognito がGoogleの認証画面にリダイレクトします。ユーザーはGoogleの画面でログインし、「ありがとうカードにプロフィール情報を渡してよいですか？」の同意画面が表示されます。

3

Google が「認可コード」を返す

ユーザーが同意すると、Googleは認可コード（Authorization Code）をCognitoに返します。これはパスワードでもトークンでもなく、「1回だけ使える引換券」です。

4

アプリが認可コード + code_verifier でトークンを取得

アプリは認可コードと、ステップ1で生成した code_verifier を Cognito のトークンエンドポイントに送ります。Cognito は code_verifier のハッシュ値が code_challenge と一致するか検証し、一致すればGoogleから取得済みのユーザー情報を元にトークンを3種類発行します（詳しくは次セクション）。

5

アプリがトークンを使ってAPI呼び出し

フロントエンドはトークンを保存し、以降のAPI呼び出しで Authorization: Bearer <token> ヘッダーに添付します。バックエンドはこのトークンを検証して、リクエストを処理します。

🔒

なぜ「認可コード + PKCE」を使うの？ — トークンを直接ブラウザに返すと、URLやブラウザ履歴に残ってしまうリスクがあります。代わりに1回限りの「引換券」を返し、トークンと交換するステップを挟みます。さらに PKCE を使うことで、認可コードを傍受されても code_verifier を持っていない攻撃者はトークンに交換できません。SPAはクライアントシークレットを安全に保持できないため、PKCEが必須です。

TOKENS

3種類のトークン

Cognito が発行するトークンは3種類あり、それぞれ役割が異なります。

📛 ID Token

「この人は誰か」の情報。

ユーザーの名前、メールアドレス、所属（@freemind.co.jp）などのプロフィール情報が含まれています。

用途：フロントエンドでユーザー名を表示したり、バックエンドで「この人は誰か」を判断するのに使います。

🎫 Access Token

「何をしていいか」の証明。

API へのアクセス権を証明するトークン。バックエンドはこのトークンを検証して、リクエストを許可/拒否します。

用途：API呼び出し時に Authorization ヘッダーに添付。有効期限は短い（通常1時間）。

🔄 Refresh Token

「トークンを更新する」ための鍵。

Access Token の有効期限が切れたとき、再ログインなしで新しい Access Token を取得するために使います。

用途：バックグラウンドで自動的にトークンを更新。有効期限は長い（通常30日）。

トークンの一生

朝、田中さんがログイン → 3種類のトークンを受け取る。
午前中にカードを送る → Access Token をAPIに添付。問題なく処理される。
1時間後、Access Token が期限切れ → Refresh Token を使って自動的に新しい Access Token を取得（田中さんは気づかない）。
30日後、Refresh Token も期限切れ → 再度Googleログインが必要。

🧩

JWT（JSON Web Token）とは：ID Token と Access Token は JWT 形式です。中身は Base64 エンコードされた JSON で、ヘッダー・ペイロード・署名の3パートからなります。署名があるため、改ざんを検出できます。jwt.io で中身をデコードして見てみると理解が深まります。

COGNITO

Amazon Cognito の役割

Cognitoは「認証の面倒なところを引き受けてくれるマネージドサービス」です。自分でOAuthの仕組みを実装するのではなく、Cognitoに任せます。

Cognitoがやること自前でやると…
Google との OAuth 連携 認可コードの交換、トークンの検証、リダイレクトURLの管理を自前で実装する必要がある
トークンの発行・検証 JWT の生成、署名、有効期限管理、Refresh Token のローテーションを自前で実装
ユーザー情報の管理 User Pool（ユーザーDB）を自前で構築・運用する必要がある
セキュリティ対策 ブルートフォース防止、CSRF対策、トークン漏洩対策を自前で実装
💰

コスト：Cognito の Lite プランは月 10,000 ユーザーまで無料です。60名のありがとうカードでは、認証コストは完全にゼロです。

Cognito の主要コンポーネント：

👥 User Pool

ユーザーの「名簿」にあたるもの。ログイン、サインアップ、パスワードリセットなどのユーザー管理機能を提供します。今回は Google OAuth と連携し、Googleアカウントを持つユーザーが自動的に User Pool に登録されます。

🔗 Identity Provider（IdP）連携

外部の認証プロバイダー（Google, Facebook, SAML等）と連携する機能。今回は Google を IdP として設定し、@freemind.co.jp ドメインのアカウントのみ許可するルールを設定します。

BACKEND

バックエンドでのトークン検証

フロントエンドから送られてくるトークンを、バックエンド（Lambda）でどう検証するかを見てみましょう。

// packages/backend/src/middleware/auth.ts // 簡略化した例（jose ライブラリで JWKS 公開鍵検証） import { createRemoteJWKSet, jwtVerify } from "jose"; const JWKS = createRemoteJWKSet( new URL( "https://cognito-idp.ap-northeast-1.amazonaws.com/ap-northeast-1_XXXXXX/.well-known/jwks.json" ) ); // Hono ミドルウェア export const authMiddleware = async (c, next) => { const token = c.req.header("Authorization") ?.replace("Bearer ", ""); if (!token) { return c.json({ error: "認証が必要です" }, 401); } try { // JWKS 公開鍵で署名検証 + issuer・有効期限チェック const { payload } = await jwtVerify(token, JWKS, { issuer: "https://cognito-idp.ap-northeast-1.amazonaws.com/ap-northeast-1_XXXXXX", }); c.set("userId", payload.sub); await next(); } catch { return c.json({ error: "トークンが無効です" }, 401); } };
🔍

検証で何をチェックしているか：① JWKS（JSON Web Key Set）の公開鍵でトークンの署名を検証（改ざん検出） ② 有効期限が切れていないか ③ 発行元（issuer）が自分の Cognito User Pool か。これらをすべてパスしたリクエストだけが処理されます。

DOMAIN RESTRICTION

@freemind.co.jp に限定する仕組み

Googleアカウントなら誰でもログインできてしまっては困ります。@freemind.co.jp のアカウントだけに制限する方法です。

方法① Cognito の Pre Sign-up Lambda トリガー

ユーザーがサインアップ（初回ログイン）しようとしたタイミングで、Lambda関数が呼び出されます。この関数でメールアドレスのドメインをチェックし、@freemind.co.jp 以外であればサインアップを拒否します。

// Pre Sign-up トリガー const email = event.request.userAttributes.email; if (!email.endsWith("@freemind.co.jp")) { throw new Error("許可されていないドメインです"); }

方法② Google Workspace の制限

Google側の OAuth 同意画面の設定で、組織内（@freemind.co.jp）のアカウントのみアクセスを許可する方法もあります。Cognito側と合わせて二重にチェックすることで、より安全です。

認可は別レイヤーで実装する

「@freemind.co.jp のアカウントでログインできた」= 全機能にアクセスできる、ではありません。
管理者のみが見られる分析ダッシュボードや、自分が送ったカードの削除権限など、認可のロジックはバックエンドのAPI側で別途実装します。 Cognitoの仕事は「この人は社員です」の確認まで。「この社員は何をしていいか」はアプリ側の責務です。

FULL PICTURE

認証フロー全体像

ログインからAPI呼び出しまでの全体の流れをまとめます。

#アクターやること
1ユーザーありがとうカードの「Googleでログイン」ボタンを押す
2フロントエンドcode_challenge を添えて Cognito の Hosted UI（認証画面URL）にリダイレクト
3CognitoGoogle のログイン画面にリダイレクト
4ユーザーGoogle でログイン ＋ 情報提供に同意
5Google認可コードを Cognito に返す
6Cognitocode_verifier を検証し、認可コード → トークンに交換。Pre Sign-upトリガーでドメインチェック
7CognitoID Token / Access Token / Refresh Token をフロントエンドに返す
8フロントエンドトークンをメモリに保存。ID Token からユーザー名等を取得して表示
9フロントエンドAPI呼び出し時に Authorization: Bearer <Access Token> を付与
10バックエンドAccess Token を検証 → 正当なら処理を実行
