01
PERSONA OVERVIEW

3人のペルソナとエコシステム

このプロダクトは、3つのタイプのユーザーが循環をつくることで成り立つ。

🌿
森山 隆志
PRIMARY — 受動的

もらったら嬉しいけど、自分からは送らない。「気恥ずかしさ」と「何を書けばいいかわからない」が壁。設計の丁寧さが最も効く相手。

☀️
中西 康弘
SECONDARY — 積極的

「面白そうじゃん」で動く。楽しいかどうかが基準。放っておいても使うが、楽しさが枯れると止まる。エコシステムの起点。

⚡
河野 真一
SUPPORTING — 懐疑的

「また横断組織の施策か」で心のシャッターが下りる。能動的には動かない。受け取る体験を壊さなければいい。

エコシステムの循環

☀️
中西さん
積極的に送る
→カードを送る
🌿
森山さん
受け取り、たまに送る
→カードを送る
⚡
河野さん
受け取る

設計の焦点：森山さんには「送る体験」のハードルを下げる。中西さんには「楽しさを枯らさない」。河野さんには「受け取る体験を壊さない」。3人へのアプローチはそれぞれ異なる。

循環を維持するために：中西さんが送る → 森山さん・河野さんが受け取る → 森山さんがリアクション or カードを送り返す → 中西さんに反応が返る。この循環が回り続けることが、プロダクトの生命線。

02
PRIMARY PERSONA

森山 隆志（もりやま たかし）

穏やかで面倒見がいい。でも、感謝を言葉にするのは気恥ずかしい。

🌿
PRIMARY — 受動的タイプ
森山 隆志

口下手だけど、後輩の面倒見はいい。心の中では「ありがたいな」と思っているが、それを文章にして送るのは気恥ずかしい。

年齢46歳
部署開発部（バックエンド）
経験年数20年以上（FREEMIND在籍12年）
家族構成妻、中学生の息子、小学生の娘
性格穏やか、職人気質、口下手だが面倒見はいい

普段の働き方

自社内で受託開発案件を担当。朝9時に出社して淡々とコードを書き、18時には退社して家族との時間を大切にしている。Chatworkは業務連絡中心に使うが、雑談チャンネルでスタンプを押すくらいの参加はしている。自分から雑談を投稿することはほぼない。

感謝に対するスタンス

後輩が助けてくれたとき、心の中では「ありがたいな」と思っている。ただ、それをわざわざ文章にして送るのは気恥ずかしい。対面で「ありがとう」と軽く言うか、Chatworkでスタンプを押す程度。「言わなくても伝わっているだろう」という感覚がある。

このアプリに対して感じること

  • もらったら素直に嬉しい。が、自分からは「何を書けばいいかわからない」
  • 100文字のメッセージを考えるのが地味にプレッシャー
  • 「コンピテンシー」という言葉に堅さを感じる。自分の言葉で「助かった」と言いたいだけなのに、大げさにされると引く
  • 仕事が忙しいときにわざわざアプリを開く動機がない

このペルソナを動かすカギ

「送る」のハードルを極限まで下げること。考えなくても、構えなくても、数秒で送れる体験。コンピテンシーはAIが裏で勝手にやってくれるので、本人は「ありがとう」と気持ちだけ書けばいい——その設計思想が伝わったとき、森山さんは動き出す。非公開で送れることが「最初の一通」を後押しする。

03
SECONDARY PERSONA

中西 康弘（なかにし やすひろ）

明るくて社交的。楽しいかどうかがすべての行動基準。

☀️
SECONDARY — 積極利用タイプ
中西 康弘

技術が好きで、人と一緒に何かやるのも好き。リーダー役を引き受けているのは義務感ではなく、楽しいから。

年齢49歳
部署開発部（フロントエンド／チームリーダー）
経験年数25年以上（FREEMINDが個人事業主の集まりだった時代から在籍、18年目）
家族構成妻、高校生の娘
性格明るい、社交的、好奇心旺盛、楽しいかどうかが行動基準

普段の働き方

技術が好きで、今でも手を動かすことに喜びを感じている。ただ、チーム内で自然とリーダー的な役割を引き受けるようになった。本人としては「偉くなりたい」わけではなく、人と一緒に何かやるのが楽しいからやっている。Chatworkでは雑談チャンネルの常連で、くだらない話にも反応するし、自分からもネタを投稿する。飲み会の幹事を自然とやるタイプ。

感謝に対するスタンス

普段から「ありがとう」「助かったよ」を口に出す人。対面でもChatworkでも気軽に言う。特に深い意味を込めているわけではなく、思ったことをそのまま言葉にしている。感謝を伝えることに照れや抵抗がない。

このアプリに対して感じること

  • 「お、面白そうじゃん」が第一印象。コンピテンシーの浸透とか理屈はどうでもいい。ツールとして面白いかどうか
  • 組織全体で一つの植物を育てる仕組みに興味を持つ。「あと何件でレベルアップだろう」とゲーム感覚で気になる
  • 自分が送ったカードに対して相手がどう反応したか気になる
  • タイムラインに自分のカードが流れて、他の人にも見てもらえるのは嬉しい。公開でどんどん送る

このペルソナを動かすカギ

中西さんは動かす必要がない。勝手に動く。問題は「動き続けてくれるかどうか」。送っても反応が返ってこない、組織の植物が成長しきってしまった、タイムラインがマンネリ化した——こういった「飽き」がモチベーションを止める。中西さんの楽しさを枯らさない仕組みが必要。逆に言えば、中西さんが楽しそうに使っている姿が、森山さんの「自分もやってみようかな」に繋がる。中西さんの体験を壊さないことが、エコシステム全体を守ることになる。

04
SUPPORTING PERSONA

河野 真一（こうの しんいち）

合理的でドライ。横断組織の施策にうんざりしている。でも、義理堅い。

⚡
SUPPORTING — 懐疑的タイプ
河野 真一

FREEMINDが個人事業主の集まりだった時代から在籍。効率至上主義で無駄が嫌い。ただし、信頼した相手には驚くほど義理堅い。

年齢45歳
部署開発部（バックエンド）
経験年数22年（FREEMIND在籍13年目）
家族構成独身、一人暮らし
性格合理的、ドライ、無駄が嫌い、信頼した相手には義理堅い

普段の働き方

長年の経験で自分のやり方が確立されている。効率を重視し、必要なことだけを最短距離でやるタイプ。かつてのFREEMINDでは自分で仕事を受けて自分で完結させる働き方が当たり前だった。その感覚が今も染み付いている。Chatworkは通知を最小限に絞っていて、自分に関係ないチャンネルはミュートしている。雑談チャンネルはたまに眺めるが、基本ROM専。

感謝に対するスタンス

感謝の気持ちがないわけではない。ただ、「仕事をきちんとやるのはプロとして当然」という価値観が根底にあり、いちいち感謝を形にすることに必要性を感じない。飲みの席で「あのときは助かったよ」と言うことはあるが、日常的にやることではないと思っている。

このアプリに対して感じること

  • 「また横断組織の施策か」が第一印象。最近入ってきた若手が主導する組織力向上フォーラムから次々と出てくる施策に正直うんざりしている。「現場のことをわかってない人たちが、理想論を押し付けてくる」という感覚
  • コンピテンシーという言葉自体、自分には縁遠い。長年コードと向き合ってきた身からすると、ふわっとしたカタカナ語で仕事を語られること自体にモヤッとする
  • 自分のペースを乱されるのが嫌。プッシュ通知が頻繁に来たら即オフにする
  • ただし、もし自分が信頼している同僚や、かつて面倒を見た後輩から「河野さんのおかげです」とカードが届いたら、表情には出さないが保存はする

このペルソナを動かすカギ

正面から「使ってください」と言っても動かない。横断組織発の施策だと知った時点で心のシャッターが下りる。河野さんが動くのは「自分宛のカードが届いた」瞬間。つまり、河野さん自身をユーザーとして獲得するより先に、河野さんに感謝している周囲の人がカードを送る状態を作ることが先。「受け取る体験」から入り口を作る。押し付けがましくない通知設計が絶対条件。

05
USER JOURNEY MAP

ジャーニーマップ：森山 隆志

「もらう → リアクションする → 眺める → 非公開で送ってみる → じんわり定着」
森山さんは急がない。ゆっくり、でも確実に馴染んでいく。

PHASE 1

認知 — 「へぇ、そんなのできたんだ」

状況

全社チャットまたは朝会で「ありがとうカード」アプリの存在を知る

行動

説明をなんとなく聞く。Chatworkの告知を流し読みする

思考

「感謝カードか…悪くはないけど」「自分が使うかと言われると…」

感情

😐フラット。興味はゼロではないが、自分から動くほどではない

設計課題

「自分が使うもの」として認識されていない。他人事として流れていく。認知フェーズで無理に引き込まなくてよい——次のフェーズ（カード受信）がトリガーになる設計にする

PHASE 2

最初のカードを受け取る — 「自分に？」

状況

ある日、メールの受信トレイに「森山さんにありがとうカードが届きました」が届いている

行動

「ん？」と思いメールを開く。リンクからアプリにアクセスし、Google認証でログイン

思考

「誰だろう」「自分に？ 何だろう」

感情

😯意外、少し嬉しい。素朴な好奇心

設計課題

ログインの手間で離脱するリスク。メールからアプリへの導線がスムーズであること、Google認証がワンクリックで完了することが絶対条件

PHASE 3

カードを読む — 「わざわざ書いてくれたのか」

状況

後輩から「先週のレビュー、丁寧に見てくれて助かりました」と書いてある。AIが「協働」のラベルを付けている。カードを開いた瞬間、パァーっと華やかな演出が入る

行動

演出に少し驚く。カードを読む。コンピテンシーラベルをちらっと見る。もう一度メッセージを読み返す

思考

「おっ」「あぁ、あのレビューのことか」「協働…まぁそういうことか」

感情

😊じんわり嬉しい。演出が気恥ずかしいけど、悪い気はしない

設計課題

この温かい気持ちを次のアクションに変換できるか。ただし「あなたも送ってみませんか？」的な誘導は不要。体験の余韻を壊さない

PHASE 4

リアクションする — 「ボタン押すくらいなら」

状況

カードを読んだ画面に、リアクションボタンが見える。自分でカードを書くほどではないが、ボタン1つなら押せる

行動

リアクションボタンを押す。「ありがとう」をワンタップで返す

思考

「これくらいなら…」「わざわざ文章書くのはアレだけど、ボタンなら」

感情

😌軽い。負担がない。これなら自分にもできる

設計課題

リアクションは「カードを書く」より遥かにハードルが低い。森山さんにとって「送る体験」への橋渡しになる。リアクションボタンはカード表示画面の自然な位置に、目立ちすぎず、でも見落とさない程度に配置する。※リアクション機能はMVPには含まれない。理想体験として記載

PHASE 5

タイムラインを眺める — 「こういう感じでいいんだ」

状況

リアクションした流れで、公開タイムラインが目に入る。中西さんが送ったカードなど、他の社員同士のカードが並んでいる

行動

いくつか読む。「こういう感じで書けばいいのか」と雰囲気を掴む

思考

「中西さんもやってるんだ」「短くていいんだな」

感情

😌安心。自分だけじゃないとわかると心理的ハードルが下がる

設計課題

タイムラインに十分なカードが流れている必要がある（初期のコンテンツ量問題）。中西さんのような積極利用タイプが初期のタイムラインを温めてくれることが前提。ベータテスト期間中に開発チーム内でカードを送り合っておく

PHASE 6

最初のカードを送る — 「まあ、気持ちが伝わればいいか」

状況

数日後、同僚が急ぎの対応を手伝ってくれた。ふと「あ、カードで送ってみるか」と思い出す

行動

アプリを開く。宛先を選び、「急な対応ありがとう、助かった」と短く書く。公開はちょっと恥ずかしいので「非公開」を選んで送信

思考

「これくらいでいいのかな」「まあ2人の間でわかればいいか」

感情

😅→😊最初は少し気恥ずかしいが、送った後はすっきりする

設計課題

「何を書けばいいかわからない」で手が止まるリスク。入力の気軽さが命。非公開を選べることで「最初の一通」のハードルが大きく下がる

PHASE 7

小さなフィードバックを得る — 「なんか育ってる」

状況

しばらくして、組織全体の植物が成長していることに気づく。「あと5件でレベルアップ」の表示が目に入る。月末に「今月のありがとうまとめ」の通知が届き、自分がもらったカードとコンピテンシーの傾向が見える

行動

植物の成長を眺める。月次まとめを読んで「協働が多いな」と思う

思考

「みんなで育ててるのか」「協働か、まぁ確かにそうかも」「あと5件か…まあ俺が送らなくても誰かが送るだろうけど」

感情

😌静かな充実感。個人のスコアではなく組織の成長だから、プレッシャーがない

設計課題

組織の植物は個人の成績ではないので、森山さんに「自分はまだ全然送っていない」というプレッシャーを与えない。それでいて「みんなで育てている」という緩やかな一体感がある。月次まとめが「また送ってみよう」のきっかけとして機能しているか

PHASE 8

定着 — 「あると悪くないな」

状況

月に1〜2回、誰かに助けられたタイミングで送る程度に落ち着く。組織の植物が少しずつ育っていくのをたまに眺める

行動

自然なペースで使い続ける。非公開で送ることが多いが、たまに「これは皆にも見てほしいな」と思ったときは公開にする

思考

「習慣ってほどじゃないけど、あると悪くないな」

感情

😌穏やかな肯定感。無理なく続いている状態

設計課題

頻度が低いので自然消滅リスクはある。月1回の集計通知が「思い出すきっかけ」として機能していれば維持できる。通知は月1回の集計＋カード受信時のみが限度

森山さんのジャーニーから見えた設計示唆

  • 受け取りが起点。自分から能動的に始めることはない
  • リアクションが「送る体験」への橋渡し。ボタン1つの軽いアクションが、カードを書く前の中間ステップになる（MVP後に追加予定）
  • 非公開が初回の鍵。「2人の間だけ」という安心感が最初の一通を後押しする
  • コンピテンシーは裏方。AIが勝手にラベル付けしてくれる。意識させすぎない
  • 組織の植物がプレッシャーなく定着を支える。個人の実績ではなく「みんなで育てている」感覚が森山さんに合っている
  • 通知設計は控えめに。カード受信通知（メール）+ 月1回の集計通知が限度

USER JOURNEY MAP

ジャーニーマップ：中西 康弘

「面白そう → すぐ使う → どんどん送る → 植物のレベルアップが楽しい → 自然と周囲を巻き込む」
中西さんはエコシステムの太陽。楽しさが燃料。

PHASE 1

認知 — 「お、面白そうじゃん」

状況

全社チャットまたは朝会で「ありがとうカード」アプリの告知を見る

行動

告知をちゃんと読む。Chatworkの雑談チャンネルで「これ面白そうじゃない？」と反応する

思考

「お、なんか新しいの出たな」「組織で植物育てるの？ どういう仕組みだろう」

感情

😄素直に興味。楽しそうかどうかで判断している

設計課題

中西さんの認知フェーズに課題はほとんどない。問題はこの後「楽しさが続くか」

PHASE 2

すぐに使ってみる — 「まずは送ってみないとわかんないよね」

状況

告知を見たその日のうちにアプリにアクセス。Google認証でログイン

行動

UIをざっと見て回る。組織の植物がまだ芽の状態なのを見て「じゃあ最初に水やりするか」と思う

思考

「まずは送ってみないとわかんないよね」

感情

😄ワクワク。新しいおもちゃを触る感覚

設計課題

初期のタイムラインが空でも中西さんは気にしない。むしろ「最初に使う人」になること自体を楽しめる。ただしUIがわかりにくいと「あれ？」となる。初見で迷わない導線が必要

PHASE 3

最初のカードを送る — 「簡単でいいね」

状況

さっそく同僚に送ってみる。昨日一緒に障害対応した後輩に「昨日の対応ナイスだったよ！」と書く。公開で送信

行動

宛先を選んで、パッと思いつくまま書いて送る。迷わない。公開を選ぶのも自然

思考

「こんな感じか。簡単でいいね」

感情

😊気持ちいい。思ったことをサッと伝えられた満足感

設計課題

中西さんにとって送信体験のハードルはほぼゼロ。AIラベリングの結果は「へぇ、まあそうかも」くらいの反応。深くは気にしない

PHASE 4

どんどん送る — 「あいつにも送っとくか」

状況

その週のうちに3〜4枚送る。チーム内の後輩、他部署で手伝ってくれた人、最近頑張っていると感じた若手

行動

思いついたタイミングでサッと送る。メッセージは短め。全部公開

思考

「あいつにも送っとくか」「タイムラインに自分のカード並んでるの、なんかいいな」

感情

😄楽しい。タイムラインが賑やかになっていくのが目に見えて気持ちいい

設計課題

中西さんが初期のタイムラインを自分のカードで温めてくれる。これが森山さんのフェーズ5（タイムラインを眺める）の体験を支える

PHASE 5

植物の成長に気づく — 「おっ、レベルアップしてる！」

状況

DESIGN IMPLICATIONS

設計示唆まとめ

3人のジャーニーから見えた、プロトタイプ設計で押さえるべきポイント。

UI / UX で必ず押さえること

  • Google認証はワンクリック。メールのリンク → 即カード表示。ここに摩擦があると森山さんも河野さんも離脱する
  • セッション保持は長めに。河野さんの2回目訪問時にログインを求めない。最低でも数ヶ月は保持
  • カード送信は最短導線。宛先選択 → メッセージ入力 → 公開/非公開 → 送信。それだけ
  • 非公開がデフォルト、または目立つ位置に。森山さんの「最初の一通」を後押しする最大の安心材料
  • コンピテンシーラベルはAIが裏でつける。ユーザーに選ばせない。意識させすぎない
  • 開封演出（パァーっとする）は受け取り体験の核。ただし河野さんのように無反応の人もいる。演出が長くなる場合はスキップ可能にする
  • 組織の植物は全員で一つ。個人の実績ではなく「みんなで育てている」感覚。送る頻度が低い人にプレッシャーを与えず、送る頻度が高い人には共有の話題を提供する
  • 「送ってみませんか？」系のナッジは入れない。河野さんを失い、森山さんも引く
  • 初見で迷わない導線。中西さんは直感的に使い始めたい。UIが複雑だと最初の勢いが削がれる
  • 通知はメール。カード受信時 + 月1回の集計のみ

やらないことリストとの関係

後でやる → 最優先

リアクション機能 — MVPには含めないが、リリース後に最優先で追加する機能として位置づける。ジャーニーマップ上では「理想体験」として記載。中西さんの送りっぱなし問題と、森山さんの送る体験への橋渡しを実現するために、MVP直後の着手が望ましい。

ジャーニーマップの読み方：本資料のジャーニーマップはリアクション機能がある前提の「理想体験」を描いています。MVPリリース時点ではリアクションがないため、中西さんのモチベーション維持は組織の植物の成長・月次集計・対面での反応で補う想定です。リアクション追加後に体験がどう変わるかの検証ポイントにもなります。

リスクと対策

初期コンテンツ問題
森山さんがタイムラインを見たとき、カードが1〜2枚しかなかったら安心感が生まれない。ベータ期間中に開発チーム内で送り合い、中西さんタイプに初期のタイムラインを温めてもらう

植物の成長上限問題
組織の植物が成長しきった後、中西さんの「次のレベル」への好奇心が消える。レベル上限の設計や、季節ごとに植物が変わるなど、MVP後に飽き対策を検討する

成功の定義を間違えない
全員がアクティブに送り合う状態は理想であって、必須ではない。河野さんのように「受け取る人」として存在するだけでもエコシステムは成立する。利用率だけをKPIにしない

継続検討事項
匿名送信の要否（非公開のみで十分か、匿名にも需要があるか）はMVP後のユーザーフィードバックで判断する
