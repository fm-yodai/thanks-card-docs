TECH FOUNDATIONS — 02

モノレポ運用ガイド

フロントエンド・バックエンド・共通型定義——すべてを1つのリポジトリで管理する。 その理由と、チームで運用するときのルールを解説します。

🏗️

アーキテクチャ設計ノートとの関連

設計ノートでは「リポジトリ構成: モノレポ（pnpm）」を採用し、フロント（Vue）・バック（Hono on Lambda）・共通パッケージ（型定義・DB・設定）・IaC（SST）・Python ETLを1つのリポジトリで管理します。

CONCEPT

モノレポとは？

モノレポ（Monorepo）は、複数のプロジェクトやパッケージを1つのGitリポジトリで管理する手法です。対になる概念がポリレポ（Polyrepo）で、プロジェクトごとにリポジトリを分けるやり方です。

POLYREPO（従来型）

📦 リポジトリを分ける

フロントエンド、バックエンド、共通ライブラリを別々のリポジトリで管理する。

github.com/company/ ├── thankyou-frontend ← リポジトリ1 ├── thankyou-backend ← リポジトリ2 └── thankyou-shared ← リポジトリ3

変更のたびに複数リポジトリ間でバージョンを合わせる必要があり、調整コストが大きい。

MONOREPO（今回の方針）

📂 1つのリポジトリにまとめる

すべてのコードを1つのリポジトリ内で、パッケージとして分けて管理する。

github.com/company/ └── thankyou-card ← リポジトリ1つ ├── apps/web # フロント ├── apps/api # バック ├── packages/shared └── infra/

1回のPRでフロント・バック・型定義をまとめて変更できる。バージョンのズレが起きない。

📌

Google、Meta、Microsoftなどの大手テック企業もモノレポを採用しています。規模が大きいから分けるのではなく、むしろ規模が大きいからこそ1つにまとめて一貫性を保つ、という考え方です。

WHY

なぜモノレポを選んだのか

ありがとうカードの特性と、チームの状況を踏まえた判断です。

🔗 型定義の共有

フルスタックTypeScriptなので、フロントとバックで同じ型定義を使います。カードの型（Card）、ユーザーの型（User）、APIレスポンスの型などを packages/shared に置けば、フロントもバックも同じ型をimportできます。ポリレポだと、npm パッケージとして公開→インストールの手間が毎回発生します。

🔄 アトミックな変更

「APIのレスポンス形式を変える」場合、バックエンドの変更とフロントエンドの変更をセットで1つのPRにできます。ポリレポだと「バック側のPRをマージ → フロント側で追従PRを作成」と2ステップになり、間にデプロイのズレが生じるリスクがあります。

🤖 AIエージェントとの相性

AIコーディングエージェント（Claude Code, GitHub Copilot）は、コンテキストが広いほど精度が上がります。フロント・バック・型定義が同じリポジトリにあれば、エージェントがプロジェクト全体を理解した上でコードを生成できます。

🛠️ CI/CDの統一

GitHub Actionsのワークフローを1箇所で管理できます。「frontendが変更されたらフロントだけビルド」「sharedが変更されたら両方テスト」といった条件分岐を1つのリポジトリ内で完結させられます。

STRUCTURE

ディレクトリ構成

ありがとうカードのモノレポ構成（想定）を見てみましょう。

thankyou-card/ # リポジトリルート ├── package.json # ルートのパッケージ定義 ├── pnpm-workspace.yaml # ワークスペース設定 ├── turbo.json # Turborepo タスク設定（任意） │ ├── apps/ # アプリケーション │ ├── web/ # Vue SPA（フロントエンド） │ │ ├── package.json │ │ ├── src/ │ │ └── vite.config.ts │ │ │ └── api/ # Hono on Lambda（バックエンド） │ ├── package.json │ ├── src/ │ └── functions/ │ ├── packages/ # 共有パッケージ │ ├── shared/ # 共通型定義・ユーティリティ │ │ ├── package.json │ │ └── src/ │ │ ├── types.ts # Card, User, Competency 等 │ │ └── constants.ts # コンピテンシーID等 │ │ │ ├── db/ # DBエンティティ・アクセス層 │ │ └── package.json │ │ │ └── config/ # 共通設定（ESLint, TSConfig等） │ └── package.json │ ├── etl/ # Python ETL（pnpm管理外・独立） │ ├── infra/ # SST（IaC） │ ├── sst.config.ts │ └── stacks/ │ └── .github/ └── workflows/ # CI/CD
📦

pnpm-workspace.yaml とは：pnpmに「このリポジトリ内のどのディレクトリがパッケージなのか」を教える設定ファイルです。これにより、apps/* と packages/* 配下の各ディレクトリが独立したnpmパッケージとして扱われ、互いに依存関係を持てるようになります。

# pnpm-workspace.yaml packages: - "apps/*" - "packages/*" - "infra"
SHARED TYPES

型の共有 — モノレポ最大の利点

フロントとバックで同じ型を使えるということが、日々の開発でどれだけ楽になるかを具体例で見てみましょう。

shared パッケージに型を定義（packages/shared/src/types.ts）：

// packages/shared/src/types.ts export interface Card { id: string; senderId: string; receiverId: string; message: string; competency: Competency; isAnonymous: boolean; createdAt: string; } export type Competency = | "sincerity" // 誠実 | "ownership" // 当事者意識 | "collaboration" // 協働 | "challenge" // 挑戦への意欲 | "creativity"; // 創造的思考

📱 フロント側で使う

// apps/web/src/components/CardList.vue import type { Card } from "@thankyou/shared"; const cards = ref<Card[]>([]); // ← Card 型が効いているので // cards[0].competency は // 必ず 5 値のどれかになる

⚡ バック側で使う

// apps/api/src/routes/cards.ts import type { Card } from "@thankyou/shared"; app.get("/cards", async (c) => { const cards: Card[] = await getCards(); return c.json(cards); // ← 同じ Card 型なので // フロントとズレない });

もし型を共有していなかったら？

バック側で competency のフィールド名を competencyTag に変えたとします。ポリレポだと、フロント側はビルドが通ってしまい、本番で初めて「APIのレスポンスが変わった」と気づくことがあります。モノレポで型を共有していれば、バック側を変えた瞬間にフロント側のTypeScriptコンパイラがエラーを出してくれます。デプロイ前にバグが見つかるわけです。

DAILY WORKFLOW

日々の開発フロー

モノレポ特有の操作を覚えておきましょう。

やりたいことコマンド補足
依存関係を一括インストール pnpm install ルートで実行。全パッケージの依存が解決される
フロントだけ起動 pnpm --filter web dev --filter で対象パッケージを指定
バックだけテスト pnpm --filter api test テスト対象を絞れるので速い
全パッケージをビルド pnpm -r build -r は再帰（recursive）。全パッケージを順に実行

⚡

--filter を覚えよう：モノレポ開発で最もよく使うオプションです。「どのパッケージに対して操作するか」を指定します。これさえ覚えれば、ポリレポと同じ感覚で開発できます。

RULES

チームで守るルール

モノレポは便利ですが、ルールなしで運用すると混乱します。以下のルールを守りましょう。

✅ やること

依存の方向を守る：apps/web → packages/shared ← apps/api の一方向。shared は他のパッケージに依存しない。

型変更は影響を確認：shared の型を変えたら、フロント・バック両方のビルドが通ることを確認してからPRを出す。

パッケージごとに責務を明確に：フロント固有のロジックを shared に入れない。バック固有のDB操作は packages/db に置く。

❌ やらないこと

ルートに直接コードを置かない：すべてのソースコードは apps/*、packages/*、infra/ 内に。ルートは設定ファイルだけ。

パッケージ間で相対パスimportしない：../../api/src/... のような参照はNG。必ずパッケージ名（@thankyou/shared）でimportする。

1つのパッケージの依存を他に漏らさない：webで使うライブラリをルートに入れない。各パッケージの package.json で管理する。

ありがちなミス：循環依存

web が shared を参照し、shared が web を参照する——これが循環依存です。 ビルドが終わらなくなったり、予期しないエラーが出たりします。
依存は必ず一方向に保つことを意識してください。迷ったら「このコードはどのパッケージの責務か？」と考えましょう。

CI/CD

CI/CDとの連携

モノレポでのCI/CDは「変更されたパッケージだけビルド・テストする」のが基本です。

STEP 1

変更検出

PRの差分から、どのパッケージが変更されたか判定

STEP 2

影響範囲特定

shared が変わったらfrontend・backendも対象に

STEP 3

ビルド・テスト

対象パッケージだけビルド＋テスト実行

STEP 4

デプロイ

影響のあるサービスだけデプロイ

🌱

学びの観点：モノレポ運用はチーム開発の基礎力を鍛えます。パッケージ間の依存関係を意識する習慣、型安全な開発の恩恵、CI/CDパイプラインの設計——これらは規模の大小を問わず、プロフェッショナルなソフトウェア開発で求められるスキルです。

覚えておきたいこと：モノレポの本質は「コードの近さ = チームの結合度」です。フロント・バックが同じリポジトリにあることで、APIの変更がフロントに即反映され、型のズレがコンパイル時に検出される。この「近さ」が、小さなチームの開発速度を大きく上げてくれます。
